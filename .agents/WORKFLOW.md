# .agents/WORKFLOW.md

> 目标：让 Codex 在商业项目中形成可控闭环：理解 → 计划 → 执行 → 验证 → 沉淀。
> 原则：减少人工干预，但不绕过关键确认点。

## 1. 任务启动检查

每次任务开始先判断：

- 用户是在询问、分析，还是要求实际修改。
- 是否已经明确目标、范围、验收标准。
- 是否涉及高风险领域。
- 是否需要读取现有项目文档。
- 是否存在能自行确认的信息。

如果只是询问或分析，不修改文件。
如果用户明确要求开发/修复/优化，可在计划后直接执行低中风险修改。

## 2. 集中提问机制

只有以下问题必须问：

- 不确认就会导致产品行为明显不同。
- 不确认就可能造成数据丢失、兼容性破坏、权限扩大或安全风险。
- 存在多个互斥实现方案，且代价或风险差异明显。
- 需要新增/升级依赖、联网、改数据库、改生产配置、改 CI/CD。
- 需要删除代码、删除文件、修改项目外文件。

提问格式：

```md
需要你确认以下问题后再继续：

1. [必须确认] ...
2. [必须确认] ...

我可以先做的无风险工作：
- ...
```

非阻塞问题处理方式：

- 做合理假设。
- 写入 `docs/codex/ASSUMPTIONS.md`。
- 最终回复中提醒用户确认。

## 3. 新项目初始化流程

触发场景：

- 用户说“初始化项目”“新项目接入 Codex”“帮我整理项目规则”。
- 当前项目没有 `docs/codex/PROJECT_CONTEXT.md`。

步骤：

1. 查看根目录、README、package/pyproject/go.mod/Cargo.toml/.sln 等配置。
2. 识别技术栈、包管理器、启动命令、构建命令、测试命令。
3. 识别主要目录和模块边界。
4. 创建或更新：
   - `docs/codex/PROJECT_CONTEXT.md`
   - `docs/codex/CODE_STYLE.md`
   - `docs/codex/DECISIONS.md`（仅在发现明确架构约束时）
5. 不改业务代码。

`PROJECT_CONTEXT.md` 推荐结构：

```md
# Project Context

## 技术栈

## 主要目录

## 常用命令

## 关键业务模块

## 运行与验证方式

## 注意事项
```

`CODE_STYLE.md` 推荐结构：

```md
# Code Style

## 命名与目录

## 组件/模块组织

## 状态管理与数据流

## 错误处理

## 测试风格

## 禁止做法
```

## 4. 功能迭代流程

步骤：

1. 明确目标与影响范围。
2. 查找相关入口：路由、菜单、组件、状态、接口、配置、测试。
3. 判断是否需要 feature flag。
4. 输出计划。
5. 执行最小必要修改。
6. 运行验证。
7. 更新 `CHANGELOG.md`、必要时更新 `DECISIONS.md`、`RISKS.md`。

功能隐藏类任务优先顺序：

1. feature flag。
2. 权限/配置控制。
3. 菜单过滤。
4. 路由拦截。
5. 条件渲染。
6. 最后才考虑删除代码，且必须先问用户。

## 5. Bug 修复流程

步骤：

1. 读取报错、日志、截图或用户描述。
2. 定位相关代码。
3. 复现或构造最小推理链。
4. 判断根因。
5. 最小修复。
6. 补充测试或说明无法补充的原因。
7. 更新 `docs/codex/BUGS.md`。

`BUGS.md` 记录格式：

```md
## YYYY-MM-DD - Bug 标题

- 现象：
- 影响范围：
- 根因：
- 修复方式：
- 回归验证：
- 后续注意：
```

## 6. 变更沉淀流程

每次修改后更新 `docs/codex/CHANGELOG.md`：

```md
## YYYY-MM-DD - 任务标题

- 类型：新功能 / Bug 修复 / UI 调整 / 重构 / 文档 / 安全
- 变更文件：
- 变更原因：
- 验证结果：
- 风险点：
- 人工验收点：
```

当做出长期设计选择时，更新 `DECISIONS.md`：

```md
## YYYY-MM-DD - 决策标题

- 背景：
- 决策：
- 理由：
- 替代方案：
- 影响范围：
- 未来回滚条件：
```

当发现问题但不适合立即修复时，更新 `RISKS.md`：

```md
## YYYY-MM-DD - 风险标题

- 风险描述：
- 影响范围：
- 当前处理：
- 建议处理时间：
- 是否阻塞发布：是/否
```

## 7. 验证策略

优先运行项目统一脚本：

```powershell
.\scripts\codex-check.ps1
```

如果失败：

1. 读取失败信息。
2. 判断是否由本次修改引起。
3. 能修则最多修 3 轮。
4. 不能修则说明阻塞点。

如果检查耗时过长或依赖缺失：

- 不自动安装依赖。
- 记录无法验证的原因。
- 给出用户本地验证命令。

## 8. 输出规则

不要输出无关长篇解释。
必须让用户知道：

- 改了什么。
- 为什么这样改。
- 怎么验证。
- 有什么风险。
- 接下来是否需要人工确认。
