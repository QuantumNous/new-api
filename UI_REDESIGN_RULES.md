# New API UI 重构规则

## 一、项目目标

当前项目仅做前端 UI 与品牌包装重构，目标是将用户可见界面包装成商业化的“昀河星泽词元运营中心”。

产品副标题：面向政企场景的一体化模型服务与AI资源运营平台。

本次改造只处理 web/default 前端，不处理 web/classic，不处理后端。

## 二、允许修改范围

允许修改：

1. web/default 下的前端页面布局；
2. 页面标题、菜单名称、按钮文案、说明文案；
3. 登录页、首页、侧边栏、顶部导航栏；
4. Logo、favicon、浏览器标题、页脚展示文案；
5. 全局样式、主题颜色、字体、间距、圆角、阴影；
6. 表格、卡片、弹窗、空状态、加载状态的视觉样式；
7. i18n/locales 中用户可见的菜单和页面展示文案；
8. 在不改变接口字段、请求逻辑、权限逻辑的前提下优化展示层。

## 三、禁止修改范围

禁止修改：

1. 不允许修改后端代码；
2. 不允许修改数据库结构；
3. 不允许修改接口地址；
4. 不允许修改接口请求参数；
5. 不允许修改接口返回字段处理逻辑；
6. 不允许修改登录、鉴权、权限判断逻辑；
7. 不允许修改计费、倍率、余额、充值、消费计算逻辑；
8. 不允许修改渠道调度、模型调用、API 转发逻辑；
9. 不允许修改 Docker、部署脚本、环境变量；
10. 不允许修改 routeTree.gen.ts；
11. 不允许随意修改 routes 中的 beforeLoad、redirect、loader 等逻辑；
12. 不允许删除 LICENSE、NOTICE、THIRD-PARTY-LICENSES 等许可证文件；
13. 不允许修改或伪造开源许可证文本；
14. 不允许删除源代码中可能涉及许可证合规的声明头；
15. 不允许把原项目版权声明伪造成我方原创声明。

用户可见界面中，不再直接展示以下字样：

- New API
- NEW API
- new-api
- One API
- QuantumNous
- GitHub Star
- 开源项目
- Powered by New API
- Built with New API
- Made by New API

用户可见产品名称统一为：昀河星泽词元运营中心。

产品副标题可保留或调整为：面向政企场景的一体化模型服务与AI资源运营平台。

对外表达重点：

1. AI资源统一运营；
2. 模型服务统一接入；
3. 算力与模型资源调度；
4. 租户化管理；
5. 应用接入管理；
6. 计费与结算；
7. 调用审计；
8. 运营监控。

## 五、菜单命名规则

将用户可见菜单和页面标题按以下方向包装：

- 渠道管理 → 模型服务通道
- Token 管理 → 词元额度管理
- API Key → 应用接入密钥
- 用户管理 → 租户与账号管理
- 充值管理 → 资源充值管理
- 消费记录 → 词元消耗明细
- 日志管理 → 调用审计中心
- 模型管理 → 模型资源池
- 系统设置 → 平台配置中心
- 分组管理 → 租户分组管理
- 令牌 → 接入密钥
- 余额 → 词元额度
- 倍率 → 计费系数

## 六、计费单位与货币展示规则

在本项目 UI 重构中，所有用户可见的计费、价格、充值、余额、消费、结算相关内容，统一使用**人民币**计价，不使用美元。

界面中的数字展示必须**明确区分「人民币金额」与「词元数量」**，避免用户把词元额度误解成钱。

### 6.1 人民币金额（必须带 ¥）

若数字代表**金额、价格、费用、单价、支付、充值、结算、余额金额**（以人民币计价的货币量）：

1. 数值前**必须**显示人民币符号 `¥`；
2. **不得**显示 `$`、`USD`、`dollar`、`美元`；
3. 币种说明文案可使用「人民币」或 `CNY`（不作为数量单位贴在纯词元数字旁）。

示例：

- `¥0`
- `¥12.50`
- `¥75 / 百万词元`（单价：人民币 / 计量单位）
- `¥1.7144 / 1M`

### 6.2 词元数量（不得带货币符号）

若数字代表**词元数量、词元额度、词元消耗、配额、Token 数、调用消耗**（系统内部的资源/额度单位，不是人民币）：

1. **不得**显示 `¥`、`$`；
2. **不得**显示 `USD`、`CNY` 等币种代码作为该数字的前缀或后缀；
3. 可显示**纯数字**，或在需要消歧时追加「词元」等单位文案（如 `24,200 词元`）；
4. 业务用语优先「词元额度」「词元消耗」等（见第五节），不要与「余额金额」混用同一套货币格式。

示例：

- `0`
- `24,200`
- `150,000,000`
- `24,200 词元`

### 6.3 区分目的与实施边界

1. 用 `¥` 明确表示**人民币金额**；
2. 用纯数字 /「词元」明确表示**资源数量**；
3. 同屏并列时，标签与格式须一致（额度列不用 `¥`，单价/金额列用 `¥`）；
4. 仅修改用户可见的展示文案与格式，**不修改**计费逻辑、后端、接口、数据库、字段名。

### 通用展示规则（与上文一致）

1. 不使用 `$`、`USD`、`dollar`、`美元` 作为主要计费单位或价格符号；
2. 金额类符号统一使用 `¥`；
3. `Token`、配额类业务表达结合产品包装，优先使用「词元额度」「词元消耗」「资源充值」「资源结算」等表述（与第五节菜单命名规则一致）。

### 禁止事项

- 不得因 UI 包装而改动金额计算、汇率换算、扣费规则或接口字段含义；
- 不得在后端或数据库中硬编码币种切换逻辑（本次仅前端展示层）；
- 不得把词元额度格式化成 `¥` 金额，也不得把人民币单价/费用格式化成无符号纯数字（除非该列明确为词元消耗量）。

## 七、视觉风格

整体采用政企级 AI 平台风格：

1. 深色科技感；
2. 蓝紫渐变；
3. 卡片式信息结构；
4. 数据驾驶舱风格首页；
5. 专业化后台表格；
6. 简洁高级，不花哨；
7. 适合政府、国企、数据公司、园区、企业客户演示。

## 八、合规边界

1. 本次目标是用户可见 UI 的商业品牌化，不是规避开源许可证义务；
2. 只处理用户可见界面层，不处理法律文件、许可证文件和第三方声明；
3. 若某处页脚、关于页或源码注释可能属于许可证要求保留的内容，必须先列为风险点，不直接删除；
4. 关于页、页脚中涉及 AGPL、上游项目致谢、许可证链接的位置，需要人工确认后再决定如何处理；
5. 不得写“完全自研”“原创底层平台”等可能造成误导的表述。

## 九、工作方式

1. 每次只改一个页面或一个组件；
2. 每次修改前先说明拟修改文件；
3. 修改后说明实际修改了哪些文件；
4. 修改后说明是否影响接口、权限、路由、计费、渠道调度逻辑；
5. 如不确定某个文件是否涉及业务逻辑或许可证合规，先说明风险，不要擅自修改；
6. 优先修改 web/default，不修改 web/classic；
7. 优先修改展示层，不改业务逻辑。
