# 硬约束

## Technology
- 后端使用 Go、Gin、GORM，结构遵循 Router -> Controller -> Service -> Model。
- 前端默认实现位于 `web/default/`，使用 React、TypeScript、Rsbuild、Base UI 与 Tailwind CSS。
- 数据库代码必须同时兼容 SQLite、MySQL >= 5.7.8 与 PostgreSQL >= 9.6。
- 后端业务代码的 JSON marshal/unmarshal 必须使用 `common/json.go` 中的封装函数。

## Security
- 绝不提交密钥、Token、Cookie、生产凭据或真实用户隐私数据。
- 不绕过鉴权、计费、安全校验、速率限制或上游服务条款。
- 计费与额度计算不得出现负扣费、溢出回绕或未审计的饱和裁剪。

## Architecture
- Relay 请求 DTO 的可选标量字段使用指针类型配合 `omitempty`，保留显式零值。
- 数据库迁移优先使用 GORM 和跨数据库写法，避免单一方言特性。
- 涉及 tiered/dynamic billing 时，先读 `pkg/billingexpr/expr.md`。
- 修改前端可见文案时必须补齐 `web/default/src/i18n/locales/` 中所有支持语言。

## Upstream Compatibility
- 严格保持上游 NEWAPI 兼容是项目硬目标；新增能力优先采用独立模块、独立路由、附加字段和可选配置，不替换上游既有行为。
- 已存在的 API 路径、请求参数、响应结构、导出列、文件名和错误语义必须保持向后兼容；需要不同合同的新能力必须新增端点，不能复用旧端点并改变含义。
- 组织账单属于只读统计层，不得修改上游预扣、结算、退款、额度、订阅、渠道累计、Relay 或日志写入链路。
- 修改共享模型、迁移入口、公共工具或上游热点文件前，必须证明独立扩展无法满足；确需修改时保持加法式、可回退，并补充兼容回归测试。
- 上游同步发生冲突时，优先保留上游安全、计费和协议语义，再把本地扩展重新落到隔离边界，禁止为减少冲突而静默改变上游合同。
