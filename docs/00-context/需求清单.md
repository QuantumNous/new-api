# 需求清单

## 核心需求

### N1 — 统一代理

统一代理多家 AI 上游服务，覆盖 40+ 提供商（OpenAI、Claude、Gemini、Azure、AWS Bedrock 等），对外暴露兼容 OpenAI 格式的统一 API。

- 支持 Chat Completions、Embeddings、Image Generation、Audio、Moderation 等主流能力。
- Relay 层按 provider 隔离适配逻辑，新增提供商不影响已有适配。
- 请求路由支持按模型、渠道、优先级和权重分发。
- 支持自动 fallback 和多渠道冗余，降低单一上游中断影响。

### N2 — 用户与鉴权

- 个人账号注册、登录（密码、OAuth、WebAuthn/Passkeys）。
- JWT 鉴权，Token（API Key）管理。
- 系统角色：普通用户、系统管理员、Root。
- 支持与企业 SSO 集成（OAuth/OIDC）。

### N3 — 计费与额度

个人账号是唯一真实扣费主体。

- 预扣 → 结算 → 退款完整链路。
- 按模型、按用量计费，支持 tiered/dynamic billing（表达式计费）。
- 额度展示支持 USD、CNY、自定义货币和 tokens 多种口径。
- 充值、兑换码、订阅等多种额度补给方式。
- 计费安全：不允许负扣费、溢出回绕或未审计的饱和裁剪。

### N4 — 组织管理

组织是"成员集合 + 报表权限 + 汇总视图"，不是付款主体。

- 管理员创建空组织，显式添加/移除成员。
- 组织角色只保留 Admin、Member。
- 一个用户同一时间只属于一个组织。
- 组织账单按成员有效期汇总现有个人消费日志，不改变扣费链路。
- 支持按成员、按模型、按渠道的用量拆分和趋势展示。
- 成员离开后，组织仍可查看其有效期内的历史消费。

### N5 — 管理后台

系统管理员通过 Web 管理后台进行全局管理：

- 用户管理：账号、余额、状态、分组。
- 渠道管理：上游渠道配置、启停、测试、优先级。
- 模型管理：模型定义、可用性、分组绑定、计费规则。
- Token 管理：查看、创建、禁用用户 API Key。
- 组织管理：组织 CRUD、成员维护、组织账单。
- 日志管理：请求日志查询、筛选、导出。
- 系统配置：站点设置、计费参数、安全策略。

### N6 — 多数据库兼容

后端同时兼容 SQLite、MySQL >= 5.7.8 与 PostgreSQL >= 9.6，不使用任何单一方言特性。

### N7 — 国际化

- 后端：go-i18n，支持 en、zh。
- 前端：i18next + react-i18next，支持 en、zh、fr、ru、ja、vi。
- 所有用户可见文案均通过 i18n 管理。

## 业务场景需求

### S1 — 企业统一接入场景

面向有 IT 治理需求的企业客户，平台作为企业内部统一 AI 网关：

- **SSO 集成**：支持企业 OIDC/OAuth 接入，员工使用企业身份登录。
- **部门与组织**：按部门/团队创建组织，划分用量和账单边界。
- **权限隔离**：不同部门可见不同模型和渠道，管理员按需配置。
- **用量审计**：管理员查看全局和部门级用量、费用、趋势。
- **额度控制**：按用户或分组设置额度上限和速率限制。
- **私有化部署**：支持客户自有基础设施部署。

### S2 — 在线 Token 服务场景

面向不需要自建平台的企业和个人开发者：

- **即开即用**：注册即获得 API Key，对接 OpenAI 兼容 SDK 即可使用。
- **多模型覆盖**：一个 Key 访问多个上游模型，无需分别管理。
- **团队账户**：通过组织功能管理团队成员和用量。
- **灵活计费**：按用量付费，支持充值和订阅。
- **高可用**：多渠道冗余、自动 fallback，确保服务连续性。
- **透明账单**：按成员、模型、渠道多维度账单展示。

## 边界需求

### B1 — 组织不侵入扣费链路

- 组织不持有余额、订阅、充值订单或 API Key。
- 加入/离开组织不改变个人余额、个人订阅、Token 归属。
- API 请求不携带或选择组织，组织归属由用户成员关系推导。
- `group` 是用户分组维度，不是组织。

### B2 — 历史账单口径

- 组织账单使用日志中已落库的 `quota` 汇总，不按当前价格重算历史。
- 金额展示参考用户侧额度展示口径（quotaDisplayType）。
- 如需正式月度账单，应另建账期快照机制（不在当前范围）。

### B3 — 安全与合规

- 绝不提交密钥、Token、Cookie、生产凭据或真实用户隐私数据到代码库。
- 不绕过鉴权、计费、安全校验、速率限制或上游服务条款。
- 客户版导出需脱敏处理（去除 content、IP、内部 request id 等）。

## 非功能性需求

### P1 — 性能

- 日志分页不全量加载，使用归并分页策略。
- 趋势聚合在数据库侧完成，不在应用内存中聚合全量日志。
- 缓存层（Redis + 内存）减少数据库热点查询。

### P2 — 可观测性

- 请求日志完整记录：模型、渠道、Token 用量、耗时、状态码。
- 回填失败时记录 SysError 后降级返回，不完全丢失数据。
- 计费饱和裁剪事件通过日志和 admin_info 审计。

### P3 — 可扩展性

- 新增上游提供商只需在 relay/channel/ 下新增适配器。
- 模型计费规则通过表达式引擎配置，不需要修改代码。
- 前端主题支持多套（default、classic），互不影响。
