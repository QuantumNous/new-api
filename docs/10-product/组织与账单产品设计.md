# 组织与账单产品设计

## 产品结论

系统采用“个人扣费、组织汇总、模型分类”的账单模型：

```text
个人账号 = 唯一真实扣费主体
组织      = 成员集合 + 报表权限 + 汇总视图
模型      = 消费分析维度，不是账号或付款主体
```

组织能力服务于企业用量治理，不改变现有个人钱包、订阅、API Key、预扣、结算和退款链路。

## 个人账单能力

个人账号持有并承担：

- 钱包额度、已用额度和请求次数。
- 充值订单、兑换码充值和管理员补单。
- 订阅套餐、订阅实例、订阅额度及额度重置。
- `subscription_first`、`wallet_first`、`subscription_only`、`wallet_only` 四种扣费偏好，默认 `subscription_first`。
- API Key 额度限制和用量统计；API Key 仍归属于个人用户。
- 消费、退款、充值和管理日志。

模型倍率、固定价格和表达式阶梯计费决定个人请求的实际 `quota`；组织报表只读取已经形成的消费事实。

## 组织产品边界

组织负责：

- 组织名称和启用/停用状态。
- 成员关系及其有效期。
- Owner、Admin、Billing、Member 四类组织角色。
- 按成员、模型、渠道、日期和日志明细汇总用量。
- 在权限范围内查看和导出组织消费日志。

组织不负责：

- 余额、充值、订阅、付款客户、发票或退款。
- API Key、Token 归属或请求时的组织选择。
- 部门、团队、项目、成本中心或跨组织切换。
- 独立组织消费流水或历史账单重算。

`group` 继续表示用户分组、模型可用性、路由和倍率，不得在产品文案或界面中解释为组织。

## 生命周期

### 创建组织

系统管理员先创建空组织，只填写名称。创建后 `owner_id = 0`，不会自动把创建人或任何用户加入组织。

### 指定 Owner 和添加成员

系统管理员在组织详情中搜索现有启用用户并添加成员。组织尚无 Owner 时，可把首位或后续成员指定为 Owner。

一个用户同一时间只能有一个活动组织成员关系。注册、登录、创建 Token、充值、订阅和 API 请求都不会自动加入组织。

### 调整角色

- 系统管理员可维护全部非锁定关系，并在组织没有 Owner 时指定 Owner。
- Owner 管理组织名称、成员和非 Owner 角色。
- Admin 管理普通成员和账单角色；当前界面不允许 Admin 修改其他 Admin。
- Billing 只查看完整组织账单和明细。
- Member 的产品目标是只查看自己在成员有效期内的用量，不查看其他成员。

Owner 不能被降级或移除。当前版本也没有 Owner 转移和组织删除流程。

### 移除成员

移除成员只结束成员关系并记录 `left_at`，不删除用户或历史日志，也不改变个人余额、订阅和 Token。

## 组织账单口径

组织账单按成员关系有效期解释个人日志：

```text
Organization -> OrganizationMember.UserId -> Log.UserId
```

- 加入组织前的消费不进入组织账单。
- 成员有效期内的消费进入组织账单。
- 离开组织后仍保留其有效期内的历史汇总。
- 默认用量视图只统计消费日志。
- 对账口径可分别纳入退款和系统调整，不能把不同类型静默合并成净额。
- 金额展示由已落库的 `Log.Quota` 按站点额度显示规则换算；当前价格只用于解释，不重算历史。

## 角色与页面权限

| 能力 | 系统管理员 | Owner | Admin | Billing | Member |
|---|---:|---:|---:|---:|---:|
| 创建组织、跨组织管理 | 是 | 否 | 否 | 否 | 否 |
| 编辑本组织名称 | 是 | 是 | 是 | 否 | 否 |
| 编辑组织状态 | 是 | API 当前允许 | API 当前允许 | 否 | 否 |
| 添加、移除成员 | 是 | 是 | 是 | 否 | 否 |
| 指定 Owner | 组织无 Owner 时 | 否 | 否 | 否 | 否 |
| 查看完整组织汇总 | 是 | 是 | 是 | 是 | 否 |
| 查看完整组织明细、导出 | 是 | 是 | 是 | 是 | 否 |
| 查看自己的组织内用量 | 是 | 是 | 是 | 是 | 产品目标为是 |

说明：后端已经把非 Owner/Admin/Billing 的账单查询限制为当前用户，但当前前端仍把 Member 挡在用量页之外；这是待对齐项，不应把它误写成已经交付。

## 第一版完成定义

- 管理员可以创建空组织、设置状态并维护成员。
- 组织无 Owner 时，管理员可以指定一名 Owner。
- 活动成员关系在 SQLite、MySQL 和 PostgreSQL 上都保持单用户唯一。
- Owner/Admin/Billing 的完整账单权限与 Member 的个人范围隔离。
- 汇总、趋势、成员、模型、渠道和明细使用同一组服务端筛选口径。
- 组织账单不触发任何预扣、结算、退款或历史重算。
- 导出复用当前筛选条件，并明确处理敏感字段。

详细的数据、接口与查询设计见 [组织与组织账单架构设计](../20-architecture/组织与组织账单架构设计.md)，页面结构见 [组织管理与组织账单 UI 设计](../90-ui-ux/组织管理与组织账单UI设计.md)。
