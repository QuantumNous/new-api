# 架构概览

## 分层结构
系统遵循 Router -> Controller -> Service -> Model 的分层结构。

## 主要模块
- `router/`：HTTP 路由入口，包含 API、relay、dashboard 与 web 路由。
- `controller/`：请求处理与接口响应。
- `service/`：业务逻辑与跨模块编排。
- `model/`：GORM 数据模型与数据库访问。
- `relay/`：AI API 代理与 provider 适配。
- `setting/`、`common/`、`dto/`、`types/`：配置、公共工具、数据结构与类型定义。
- `web/default/`：默认前端。

## 组织扩展边界

详细设计见 [组织与组织账单架构设计](./组织与组织账单架构设计.md)。

组织模块是现有个人账号体系之上的无侵入扩展。

```text
个人扣费链路：请求 -> Token -> User -> 个人钱包/订阅 -> Log
组织汇总链路：OrganizationMember.UserId -> Log.UserId -> 组织报表
```

组织模块只维护：

- `Organization`：组织基本信息。
- `OrganizationMember`：成员、角色和成员有效期。
- 组织报表权限。
- 按成员有效期汇总的组织账单视图。

组织模块不进入 relay、Token 鉴权、预扣、结算、退款、充值和订阅链路。现有 API 请求不会携带或选择组织，组织归属只由当前登录用户的有效成员关系推导。

## 接口分层

- 用户侧组织接口使用 `current` 语义，后端根据当前用户唯一有效成员关系找到组织。
- 管理员组织接口使用显式组织 ID，供系统管理员跨组织维护和排查。
- 组织账单接口只读现有日志，不写账务数据。
