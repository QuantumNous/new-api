# 组织客户账单展示实施方案

## 背景

`/admin/organizations/:id` 当前的 `Billing` Tab 只展示组织总账摘要，缺少面向客户对账所需的成员拆分和模型拆分。作为给客户的组织账单，页面必须能回答三类问题：

- 本账期组织总共用了多少。
- 每个成员分别用了多少。
- 每个模型分别产生了多少用量和费用。

本轮实施只补齐展示和前端接口接入，不改变现有扣费链路。个人账号仍是唯一真实扣费主体，组织只是成员集合、报表权限和汇总视图。

## 当前可复用基础

后端已经暴露组织账单维度接口：

- `GET /api/admin/organizations/:id/billing/summary`
- `GET /api/admin/organizations/:id/billing/members`
- `GET /api/admin/organizations/:id/billing/models`
- `GET /api/admin/organizations/:id/billing/channels`
- `GET /api/admin/organizations/:id/billing/trend`
- `GET /api/admin/organizations/:id/billing/logs`
- `GET /api/admin/organizations/:id/billing/logs/export`

用户侧组织账单页已经接入了 summary、trend、models、channels，但管理员组织详情页当前只接入 summary 和 logs。

后端账单口径已经按组织成员有效期过滤：

```text
log.user_id = organization_member.user_id
log.created_at >= organization_member.joined_at
left_at = 0 or log.created_at < organization_member.left_at
```

这保证了离开组织的成员仍能在其历史有效期内被计入组织账单。

## 实施范围

### 本轮做

1. 前端 API 层补齐管理员组织账单维度接口。
2. 管理员组织详情页 `Billing` Tab 展示：
   - 总账摘要。
   - 使用趋势。
   - 按成员用量账单。
   - 按模型用量账单。
   - 按渠道用量账单。
3. 维度表格补齐客户账单关键字段：
   - 金额，沿用用户侧额度展示口径。
   - 原始 quota，用于审计和排障。
   - 请求数。
   - 输入 tokens。
   - 输出 tokens。
   - 总 tokens。
   - 占比。
   - 模型计费规则摘要。
4. 刷新按钮同时刷新总账、趋势、成员、模型、渠道数据。
5. 所有新增 UI 文案进入 i18n。

### 本轮不做

- 不新增组织余额。
- 不新增组织充值。
- 不新增组织订阅。
- 不新增组织 API Key。
- 不重算历史账单。
- 不把当前模型价格当作历史结算价格。
- 不重构组织后端聚合实现。
- 不改客户版脱敏导出；CSV 脱敏和多 sheet 导出后续单独做。

## 页面设计

管理员组织详情页保留现有一级 Tabs：

```text
Members | Billing | Logs
```

`Billing` Tab 内采用紧凑后台账单布局：

```text
Date range | User ID | Model | Channel ID | Refresh | Export

Summary:
  Requests | Amount | Raw Quota | Prompt tokens | Completion tokens | Active members

Tables:
  Usage trend
  Member usage
  Model usage
  Channel usage
```

成员账单表：

```text
Member | Amount | Raw Quota | Share | Requests | Prompt tokens | Completion tokens | Tokens
```

模型账单表：

```text
Model | Amount | Raw Quota | Share | Requests | Prompt tokens | Completion tokens | Tokens | Pricing
```

渠道账单表：

```text
Channel | Amount | Raw Quota | Share | Requests | Prompt tokens | Completion tokens | Tokens
```

## 数据口径

- 总账、成员账、模型账、渠道账必须使用同一组筛选参数。
- 拆分表的 `Raw Quota` 以日志中已经落库的 `quota` 汇总为准。
- `Amount` 不重算价格，直接参考用户侧额度展示方式：以 raw quota 为输入，按站点 `quotaDisplayType` 显示为 USD、CNY、自定义货币或 tokens。
- 模型表的 `Pricing` 仅用于解释当前配置，不作为历史账单重算依据。
- `Share` 使用当前筛选窗口下 `row.total_quota / summary.total_quota` 计算；总额为 0 时显示 `0%`。
- tokens 展示 `prompt_tokens + completion_tokens`，但仍保留输入/输出拆分。
- 日期筛选和趋势按 UTC 账单日处理，避免浏览器本地时区和服务器时区差异导致账期漂移。

## 性能与正确性修复

- 日志分页不再对每个成员直接拉取 `startIdx + pageSize` 条记录；改为先取每个成员的首条记录，再按全局时间顺序归并，并只对实际贡献当前页的成员继续批量取数。
- 趋势不再把日志全量拉回 Go 内存按天聚合；改为按成员窗口在日志库侧按 Unix day bucket 聚合，再在 Go 中合并同一天结果。
- 用户名、渠道名回填失败时记录 `SysError` 后降级返回，避免账单字段缺失时完全无运维信号。
- 维度结果中的 `Pricing` 是响应扩展字段，显式标记为非 GORM 字段，避免聚合 `Scan` 被误判成关联关系。

## 验收标准

- 进入 `/admin/organizations/1` 的 `Billing` Tab 后，能同时看到总账、趋势、成员账、模型账和渠道账。
- 修改日期、成员、模型或渠道筛选后，各区块使用同一筛选条件。
- 刷新按钮会刷新所有账单区块。
- 导出按钮保留现有明细导出行为。
- TypeScript 类型检查通过。
- 前端 lint 不引入本次变更相关 error。
- i18n 同步后新增文案在所有前端语言包中存在。

## 后续事项

- 增加客户版脱敏导出，去掉 `content`、IP、内部 request id、上游 request id 等敏感字段。
- 如要做正式月度账单，应新增账期快照，固化账单生成时的汇总、拆分和价格解释，避免历史账单随配置变化漂移。
- 如果组织未来成为真实付款主体，需要另起架构方案，引入独立 BillingAccount，而不是复用本轮报表型组织账单。
