Harness Engineering · Internal Tech Share · 2026

不是把活交给 AI，
而是为 AI 工程化它的工作环境

本文以生产项目 flatkey-newapi（聚合 40+ 上游 AI 供应商的 API 网关，Go + React，跑在 GCP Cloud Run）为样本，论证一个观点：AI 时代真正要做的工程，是 harness engineering——围绕 agent 把"能读什么、能做什么、被什么约束、靠什么验证、如何沉淀"全部工程化。开发-发布提速、新人当天产出、运维一句话搞定，都是这套 harness 的结果，不是模型本身的魔法。

Thesis 模型是商品，harness 是护城河 GCP Cloud Run · Cloud SQL · Redis CI/CD GitHub Actions + WIF 月成本 ≈ $215–245

★引言：把这一切叫做 Harness Engineering

同一个模型，给它一个空仓库，它只能瞎猜；给它一套工程化的 harness，它能当天产出合规代码、一句话完成生产运维。差距不在模型，在 harness。

"Harness"是 agent 运行所处的整个脚手架。Harness engineering 就是有意识地去建造它——而不是反复调 prompt。一套好的 harness 有五层，本文接下来每一节，都是在填其中一层：

PERCEIVE
① 感知层
让 agent 读得懂：以根 CLAUDE.md 为总入口/目录 → 业务地图（GitNexus 4.2 万符号）、45 份按需加载的 AGENTS.md、核心业务 md、项目记忆  →  §③ §◎ §④
ACT
② 行动层
让 agent 做得了：CLI（gcloud/gh/terraform）、封装成 skill 的能力、CI workflow  →  §① §◎
CONSTRAIN
③ 约束层
让 agent 不犯错：CLAUDE.md 硬规则、改前影响分析、写审分离、反 Slop 闸门、发布人工 Approve  →  §① §③
VERIFY
④ 反馈层
让 agent 知道对没对：go test/bun build、健康检查 /api/status、金丝雀、verifier 证据  →  §①
REMEMBER
⑤ 记忆层
让经验产生复利：项目记忆文件、运维 playbook、踩坑写回  →  §④

下面四节是这五层的展开；最后在 收束 处把它们合回这张 harness 全景图。

1发布：一条 AI 能亲手驱动的流水线

这一节的重点不是流程多严谨，而是：从开 PR 到上线再到回滚，每一步 AI 都能用 gh / gcloud 命令亲手驱动或查询。人只在一个地方出现——发布前点一下 Approve。其余全自动、用户无感。

1
合入 main AUTO
AI 开 PR → 过反-Slop 质量闸门 → 合并；之后整条链路自动接管
2
自动构建镜像 → Artifact Registry AUTO
WIF 短期令牌，无长期密钥；多 tag 可精确回溯
★
人工 Approve —— 唯一的人工闸门 MANUAL
挂在 GitHub production 环境，邮件通知 reviewer，网页点一下放行
4
新 revision 不接流量 → 健康检查 → 金丝雀 10% → 100% AUTO
老版本持续服务，出问题在 10% 阶段就发现，全程用户无感
这条链路 AI 全程命令行可驱动（人只保留 Approve）：
  • 提交：gh pr create 开 PR；gh run view --log-failed 查 CI 为什么挂
  • 发布：合 main 即自动；gh workflow run gcp-deploy 也能手动触发（可指定已有镜像 tag 跳过构建）
  • 确认：gcloud run services describe 看当前 revision 与流量分配，验证是否真生效
  • 回滚：gcloud run services update-traffic ...=100 约 10 秒切回任意旧版（GitHub 上还有 GCP Rollback workflow 留痕）

2基础设施：一朵 AI 能直接操作的云

整套生产环境（Cloud Run + Cloud SQL + Redis + HTTPS LB + Secret Manager）全部用 Terraform 代码描述，一条 terraform apply 可从零重建；CI 用 WIF 短期令牌，0 把密钥落地。但分享的重点不是这套架构有多标准，而是：这朵云 AI 能直接查、直接调、直接建——不用点控制台、不用记资源名。

AI 通过 CLI 直接操作这朵云：
  • 查：gcloud run services describe 看流量 · gcloud logging read 拉日志 · gcloud sql instances describe 看库状态
  • 调：update --min-instances=N 扩容 · update-traffic 切流量/回滚 · --update-env-vars 改配置
  • 建：terraform plan/apply 加域名 / 升 HA / 加 secret，全部 IaC 走 PR review
新人不需要懂全部资源、不需要点 GCP 控制台——说一句话，AI 用 CLI 替你操作（实战见 §◎ 一句话运维）。
而且 AI 操作时会自动规避坑（因为写进了项目记忆）：Cloud Run 流量是 revision-pinned 的——gcloud 改完配置必须手动 update-traffic 否则不生效；部署 SA 最小权限导致 terraform apply 暂走不通 CI。这些都已沉淀为记忆，下一个人/agent 不会重踩（见 §④）。

3基于 AI 如何做到快速开发、发布上线

AI 不是"写几行代码"那么简单——真正的杠杆在于把整个仓库变成 AI 能精确导航、能安全改动、能自我验证的环境。而这一切的起点，是一个文件。

⓪ 一切的总入口：根 CLAUDE.md

会话启动时，只有根 CLAUDE.md / AGENTS.md 会自动进入上下文，其余 45 份子文档一个都不加载。所以这唯一被"常驻"的文件，决定了 agent 和新人对整个项目的第一印象和导航能力。它的设计哲学很关键：它是一份"目录 / 路由表"，不是一本"全文手册"。

CLAUDE.md ← 唯一常驻上下文（总入口 / 目录） │ ├─ 规则 Rule 0–8（硬约束，常驻） │ Rule 6 ─→ 读 pkg/billingexpr/expr.md （计费表达式，按需） │ Rule 7 ─→ 读 deploy/gcp/docs/OPERATIONS.md （GCP 运维，按需） │ Rule 8 ─→ 读 relay/channel/task/AGENTS.md （seedance SOP，按需） │ └─ 模块级文档索引（45 份 AGENTS.md，全部"按需才读"） controller/ service/ model/ router/ middleware/ relay/ relay/channel/ relay/channel/task/ … setting/* pkg/* dto/ types/ oauth/ i18n/ web/* ▲ 改哪个模块，才去 Read 哪一份 —— 不一次性全读

CLAUDE.md 里把这条"按需加载"明确写成了给 AI 的阅读规则：

禁止一次性把全部 AGENTS.md 读入上下文（浪费 token）；
必须在「准备改某模块代码前 / 需要了解其内部约定 / 跨模块协作 / 用户提问指向该目录」时，才主动 Read 对应那一份；
可省略：若 Rule 1–8 已覆盖（如纯 JSON / 跨 DB / Bun 问题），不必再读子文件。
为什么"目录式入口"是 harness 的精髓：它让常驻上下文极小且稳定（一份索引 + 一批硬规则），同时让全部知识可达且精确（每条规则、每个模块都标好了"去哪读"）。新人 / agent 不必先吞下整个仓库，而是读一个文件就拿到全局地图 + 行为准则，再顺着指针即时加载需要的那一块。这就是感知层（Perceive）的路由器——用最少的常驻 token，撬动最大的知识可达性。

① 它索引的下一层：分层代码地图 + 显式规则

仓库为每个主要模块维护一份 AGENTS.md（30+ 份），记录该目录的职责、关键文件、内部约定、测试方式。约定是按需加载——不一次性塞进上下文浪费 token，改哪个模块就读哪份。根 CLAUDE.md 还固化了一批硬规则：

规则内容为什么
Rule 1JSON 必须走 common.Marshal/Unmarshal一致性 + 未来可换更快的 JSON 库
Rule 2SQLite / MySQL / PostgreSQL 三库同时兼容用 GORM 抽象，禁单库专属语法
Rule 5relay 请求 DTO 可选字段用指针 + omitempty否则零值 0/false 会被静默丢弃
Rule 8新 seedance 视频渠道照 SOP 走统一入参契约 + 白标合规（详见 §4）
这些规则让 AI 和新人第一次提交就符合规范，而不是 review 时被打回。规则即护栏。

② 多智能体协作 + 代码图谱 + 自我验证

  • 代码图谱（GitNexus）：4 万+ 符号、13 万+ 关系建索引。改任何函数前先跑影响分析（blast radius），HIGH/CRITICAL 风险会预警——改之前就知道会牵动谁，而不是上线后才发现。
  • 专职 agent 分工：探索、规划、执行、审查、验证分别由不同 agent 承担，关键约定是"写"和"审"分属不同上下文——不自我背书，由独立的 reviewer/verifier 把关。
  • 证据优先：声称"完成"前必须有验证证据（测试通过、健康检查 200）。
真实战果：在一次"对接 BlockRun seedance 渠道"的极限 review 中，AI 审查抓到一个隐蔽 bug——relay_task.go 在调 DoResponse 前就拦截了所有非 200 响应，导致上游返回 202 Accepted 的异步渠道端到端不可用，而且每次提交都白白扣费。修复方案（在 DoRequest 里把 202 归一成 200）当场沉淀为项目记忆 task-adaptor-202-gate，下一个接异步渠道的人直接受益。

③ 把"发布"也交给机器

§1 的整条流水线意味着：AI/开发者写完代码、本地验证、合 main，剩下构建-健康检查-金丝雀-放量全自动。人只在两处出现：PR review 背书 和 一次发布 Approve。开发到上线的周期被压到天级甚至小时级。

④ 核心业务"有文档"，关键操作"被封装"

效率的另一半来自两层资产化：核心业务都配了一份"动手前必读"的 md 文档，高频/高危操作被封装成可重复执行的单元（skill / workflow / playbook）。AI 和新人不靠口口相传，照着调用即可。

核心业务 = 一份 md + 一条硬规则

根 CLAUDE.md 把"读文档"写成了强制规则，每条规则直接指向一份核心业务文档：

核心业务必读文档硬规则
计费表达式 / 动态定价pkg/billingexpr/expr.mdRule 6：动表达式系统前必读，讲清语言、架构、token 归一、版本化
GCP 基础设施操作deploy/gcp/docs/OPERATIONS.mdRule 7：跑任何 terraform/gcloud 前必读（state 位置、双认证、流量陷阱…）
新增 seedance 视频渠道relay/channel/task/AGENTS.md SOP + docs/api/seedance-video-api.htmlRule 8：照 SOP 与 kuaizi 样板做，保证契约统一 + 白标合规
全仓库 30+ 份 AGENTS.md 构成分层代码地图，外加 expr.md / 白标 API HTML / 三份 GCP 手册等专题文档——核心业务没有一处"只在某人脑子里"。

关键操作 = 可调用的封装单元

  • 代码智能封装成 skill：GitNexus 把"探索架构 / 影响分析 / 调试 / 重构 / 重命名"封装成 6 个可直接调用的 skill（gitnexus-exploring、gitnexus-impact-analysis、gitnexus-debugging、gitnexus-refactoring…）。新人不用学一套工具，说"评估改这个函数的影响"即可触发。
  • 发布/回滚封装成 workflow：gcp-deploy.yml / gcp-rollback.yml 就是"部署"和"回滚"两个操作的封装——参数化（指定 image_tag / revision）、可重复、带审批与审计，等价于一个"部署 skill"。
  • 运维封装成 playbook：轮换密钥、加域名、扩容、排障，全部在 DEPLOYMENT.md / OPERATIONS.md 里写成可照抄的步骤序列。
"文档化 + 封装化"的复利：业务有 md，新人知道"为什么这么设计"；操作有封装，新人不需要懂全部细节也能安全地执行。两者叠加，AI 能自动调用、新人能照单执行——这正是"快"的底层原因。

◎业务地图：新人接手 + AI = 当天就能产出

前面三节的能力最终都汇聚到一件事上——业务地图。它是让"陌生人当天写出符合规范的真实改动"成为可能的底座：AI 靠它精确定位，新人靠它跳过"摸几周代码"的阶段。

新人 Day 1：从 0 到产出一个真实改动

以"接入一个新的 seedance 视频渠道"为例——一个对仓库零认知的新人，配合 AI 和业务地图，路径是这样的：

1
问地图，不读代码
gitnexus_query("video task channel") → 定位到视频任务的执行流和关键文件，几秒钟拿到"该看哪里"
2
读语义地图拿范式
打开 relay/channel/task/AGENTS.md 的 SOP + 参考实现 kuaizi/——知道"只写私有映射、不改入参契约、必须白标"
3
改之前先算影响
gitnexus_impact({target, direction:"upstream"}) → 看清 blast radius，HIGH/CRITICAL 会预警，避免误伤
4
AI 抄样板写映射
照 kuaizi 写 build<Channel>CreateRequest 映射 + 取值校验 + 注册，符合规则一次过
5
提交即上线
PR → 反 Slop 守门 → 合 main → 自动 build → Approve → 金丝雀放量（见 §1）
结果：新人不需要先"读懂整个系统"，只需要"在地图上找到自己那一块"。理解范围从"全仓库"收敛到"一个模块 + 一条执行流"，配合 AI 当天就能产出合规代码。

地图之外：CLI 让 AI 直接动手

光看得懂还不够——业务地图解决"读"，CLI 工具解决"做"。AI 不只在编辑器里改代码，还能通过命令行直接操作云与仓库，把"查问题→改→验证→发布"连成闭环：

CLIAI 拿它做什么例子
gcloud查/操作 GCP：看活跃 revision、读 Cloud Run / SQL 日志、切流量回滚、改 env、查证书状态gcloud run services describe 看流量分配；gcloud logging read 定位 502；update-traffic 回滚
gh查/操作 GitHub：看 PR 与 CI 状态、读失败日志、触发 workflow、查 run 历史gh run view <id> --log-failed 看构建为何挂；gh run list 查发布历史
terraform基础设施变更：plan 看 diff、apply 落地（state 在 GCS）加域名 / 升 HA / 加 secret，全部 IaC 走 PR review
bun · go本地构建/测试/i18n 同步，产出验证证据bun run build、go test、bun run i18n:sync
读（地图）+ 做（CLI）+ 沉淀（记忆）三件套：AI 用 GitNexus 看懂系统、用 gcloud/gh 直接排障与发布、把新发现写回项目记忆。一个排障到修复的回合，可以全程在对话里闭环完成——这正是"快"从感觉变成机制的地方。
边界仍在人手里：能用 CLI 直接操作生产，不等于放任。发布仍卡 production 人工 Approve、改生产前先看清影响、不可逆操作先确认——CLI 放大的是效率，不是越权。

一句话运维：你说人话，AI 跑对命令

最能体现"记忆 + CLI"威力的，是日常运维。你不需要记任何 gcloud 命令，也不需要知道有哪些坑——因为项目记忆和手册里已经写好了。你说一句自然语言，AI 自动翻成正确的命令、还会主动补上那些"不看记忆就会漏"的隐性步骤。

你：查一下线上 flatkey-newapi 为什么有点 502
AI 自动执行 ▾（依据 DEPLOYMENT.md 故障排查章节）
gcloud logging read "resource.type=cloud_run_revision AND \
  resource.labels.service_name=newapi" \
  --limit=50 --order=desc

不用你提供 project / region / 服务名 / 过滤语法——记忆里都有，AI 直接拼好查询。

你：高峰快到了，把线上实例数先提到 5
AI 自动执行 ▾（两步，关键在第二步）
gcloud run services update newapi --region=us-west1 --min-instances=5
# ↓ 因为记忆 cloud-run-traffic-revision-pinned，AI 主动补这一步
gcloud run services update-traffic newapi --region=us-west1 \
  --to-revisions=<新建的-revision-name>=100

这就是记忆的价值：不看记忆的人改完 --min-instances 就以为完事了——可 Cloud Run 流量是 revision-pinned 的，新 revision 不会自动接流量，扩容根本没生效。AI 因为记着这条坑，自动补上 update-traffic。

你：刚发的版本有问题，回滚到上一个
AI 自动执行 ▾（约 10 秒生效）
gcloud run revisions list --service=newapi --region=us-west1 --limit=5
gcloud run services update-traffic newapi --region=us-west1 \
  --to-revisions=<上一个-revision>=100

先列出历史 revision 确认目标，再切流量——步骤顺序也来自手册，不会切错。

从"要记一堆命令"到"说一句话"：查日志、扩缩容、看流量分配、回滚、查证书状态、轮换密钥……这些过去需要专人记忆、容易出错的运维操作，现在沉淀在记忆与手册里 → AI 读取 → CLI 执行。新人不必先成为 GCP 专家，也能安全地运维生产——而且 AI 比人更不会漏掉那些隐性步骤。

4项目记忆的重要性：让新同事"几小时上手"

代码告诉你"系统长什么样"，但很多关键知识不在代码里——为什么这么设计、踩过哪些坑、上线流程的隐性约定、某个渠道的对接范式。这些就是项目记忆。它们以两种形态沉淀：仓库里的 AGENTS.md / docs/，以及 AI 的持久化记忆文件。

记忆长什么样：四条真实记忆

project · seedance-channel-onboarding-sop

为什么这对新同事是降维打击

场景没有项目记忆有项目记忆
接一个新视频渠道读几千行 relay 代码，猜入参该怎么设计，可能发明出不一致的格式读 SOP + 抄 kuaizi，半天搞定，天然白标合规
改 Cloud Run 一个开关改了 TF 合 main，以为生效，其实没上线，排查半天记忆直接告诉你走 gcloud + update-traffic
发布一个版本不知道卡审批、不知道金丝雀、不敢点照 DEPLOYMENT.md 三步走，回滚也有一条命令

本质：项目记忆把"只有原作者脑子里有"的隐性知识，变成仓库里可检索、AI 可加载、新人可执行的显性资产。换人、扩团队、交接，成本趋近于零。

第二个典型案例：flatkey

另一个生产项目 flatkey（sub2api 的二开 fork）把这套做到了极致——它的 docs/ 目录本身就是一座运维知识库：

  • 部署/升级 playbook：lb-upgrade-playbook.md、fleet-dashboard-deploy-playbook.md、cf-subdomain-vercel-playbook.md
  • 线上排障：errors.md、dogfood-log.md、morning-brief-*.md（每日运行简报）
  • 账号安全与调度：fleet-account-safety-playbook.md、upstream-account-safety.md、「账号调度策略」
  • 新人 onboarding：onboarding.md、sato-onboarding-guide.md——直接照着就能接手
  • 甚至已有多份 HTML 分享文档（拓扑、架构 review、路由方案卡片），知识天生可传播

再加上 AI 记忆里记着的硬知识，比如「macOS pkg 在指定机器打包签名；2026-05 因旧证书被吊销换成 Solvea AI Inc. 的 Developer ID」——这种"换证书"级别的运维细节，正是交接时最容易丢、丢了最致命的东西。

✦收束：Harness Engineering 全景

把全文合回开篇那五层——每一层都有仓库里看得见摸得着的实物在支撑。这就是 harness engineering：不是抽象方法论，而是一堆可以今天就照着建的具体资产。

Harness 层解决什么本项目里的实物
① 感知 Perceiveagent / 新人读得懂系统GitNexus 图谱（4.2 万符号 / 13.5 万关系 / 300 执行流）· 45 份 AGENTS.md · expr.md 等核心业务文档 · 项目记忆
② 行动 Actagent 能直接改与发gcloud/gh/terraform/bun/go CLI · GitNexus 6 个 skill · gcp-deploy/gcp-rollback workflow
③ 约束 Constrain放大效率不放大风险CLAUDE.md Rule 0–8 · 改前 gitnexus_impact · 写审分离 · pr-check 反 Slop · production 人工 Approve
④ 反馈 Verify声称"完成"前有证据go test/bun build · 健康检查 /api/status · 金丝雀 10%/3min · verifier 独立验收
⑤ 记忆 Remember经验复利、换人零成本项目记忆（流量陷阱 / 202 gate / env 双来源 / IAM gap）· 三份 GCP 手册 · flatkey docs/ playbook 库
飞轮：感知层让 agent 看懂 → 行动层让它动手 → 约束层兜住边界 → 反馈层确认结果 → 记忆层把新经验写回感知层。每跑一轮，harness 自己变得更强，下一个人（或下一个 agent）起点更高。

AI 时代真正的护城河，不是"谁的模型更强"——模型大家都能调到。护城河是 harness engineering：谁把自己的业务、基础设施、运维知识工程化成 agent 能感知、能行动、有约束、可验证、会沉淀的环境，谁就能让"快速开发、快速上线、快速接手"从英雄主义变成系统的默认属性。
