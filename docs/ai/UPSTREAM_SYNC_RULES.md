# Upstream 同步规则

> 本文件只在“同步官方更新”类任务中使用。

这样做符合 `YAGNI` 和 `SOLID`，直接收益是把“同步上游”和“开发业务功能”拆成两类任务，降低冲突与回归风险。

## 同步原则

1. 官方 upstream 变更必须先进入集成分支，再考虑进入业务开发分支。
2. 同步 upstream 的提交 / PR 不允许顺手夹带业务功能改动。
3. 同步 PR 与业务 PR 必须分开。
4. 同步前必须先阅读 `AGENTS.md` 和 `AI_TASK_TEMPLATE.md`。

## 同步任务模板约束

在开始同步前，必须明确：

- 上游来源与目标分支
- 本次同步范围
- 是否涉及数据库、DTO、配置或前端构建链变化
- 本次同步完成后的回归验证方式

## 推荐流程

1. 获取 upstream 最新变更
2. 合并 / rebase 到集成分支
3. 解决冲突时保持项目受保护信息不被修改
4. 运行最小必要验证
5. 单独提交同步 PR

## 禁止事项

- 禁止在同步 upstream 时顺手实现业务需求
- 禁止把同步任务和功能开发揉成一个提交
- 禁止在未说明验证方式的情况下直接合并上游改动
