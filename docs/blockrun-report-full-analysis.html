② 我方 new-api 可解决的

A1 · Error leakage（P0）— 白标脱敏 代码 · 已实现待部署

失败响应走 service.RelayErrorHandler，把上游结构化错误的 message 原样透传；泄露文本同时存在于 NewAPIError.Err（供 ToClaudeError）和 RelayError.Message（供 ToOpenAIError）两处。

// 上游回传、被我方透传给客户端的内容：
GenerateOpenAIRequest.max_tokens of type x not present in request id
// 该字符串 grep 全库为空 → 非我方代码生成

已实现方案（整段替换 + 白标 gating）：

  • types.NewAPIError.OverrideMessage()：同时改写 Err 与 RelayError 两处 message（仅改 Err 会漏 OpenAI 渲染路径）。
  • service.ScrubWhitelabelError()：仅对白标渠道(BlockRun 100/101/102) + 命中泄露特征(CamelCase *Request. / *.go:行号 / not present in request / panic: 等)时，整条 message 换成通用文案，原文写服务端日志。
  • 接线在 controller/relay.go 唯一渲染 chokepoint（非高风险共享的 RelayErrorHandler，gitnexus impact=HIGH/11 caller）→ 40+ 其他渠道零影响。
  • 普通报错(限流/内容策略)与非白标渠道原样透传。已加单测 TestScrubWhitelabelError，build/vet/回归全绿。

A2 · 鉴权 402（D1）— 状态码改写 配置

ResetStatusCode + 渠道 status_code_mapping 机制在 claude 路径已接好（relay/claude_handler.go:209）。给该渠道配 {"402":"401"} 即可把上游 x402 的 402 在出口改写成标准 401，既不泄露「支付门控代理」性质，又符合鉴权语义。纯配置，无需发版。

A3 · "Claude Official" 标签 — 已作废

经澄清：该渠道确为真官渠，"Claude Official" 仅是分组标签，非对上游原生性的声称。此项无需任何操作。

③ 必须抛给 BlockRun 上游的

B1 · 后端泄露内部报错（root cause of P0 #1）上游

GenerateOpenAIRequest 是 BlockRun 后端自己的 Go 结构体。源头不吐内部实现名，得他们改。我方只能在出口「抹」。

B2 · 标准 max_tokens 请求功能性失败（root cause of P0 #2）上游

max_tokens ... not present in request 意味着上游连标准 max_tokens 字段都没正确处理、直接内部报错。这不是「错误信息难看」，是请求本身处理失败——真实客户端发同样请求照样拿不到结果。我方脱敏只让它不泄露、不解决可用性。

B3 · BlockRun 代理/校验层用 OpenAI 结构体中转（B2 的机制）上游

注：渠道已确认为真官渠，这里不是原生性/真伪问题。我方是 native 透传（零转换），上游却吐 GenerateOpenAIRequest，说明 BlockRun 在官方 Claude 上游前面还有一层自己的代理/校验层（很可能为 x402 计费、参数校验），该层内部用 OpenAI 格式结构体处理请求，并在标准 max_tokens 上失败——这正是 B2 功能性失败的机制。需上游修复该代理层，使标准 Anthropic 字段端到端正确处理。

B4 · x402 / 402 鉴权语义 上游协议

402 Payment Required 源自 BlockRun 的 x402 微支付协议（relay/channel/blockrun/x402.go）。是否在鉴权失败时返回更标准的码，是上游协议决定；我方只能改写、不能改其协议。

B5 · 模型质量 / 疑似降级（D2）上游

no-extra-text 45、constraint 63、arithmetic partial、code understanding 5 过 3。我方零替换，故这些纯属上游模型实际输出。对真 sonnet-4.5 偏低，需上游核实模型指纹、确认无量化/替换。

④ 无法解决 / 只能检测与决策的

  • 残留风险 我方两个 P0 修复本质是治标。同一个失败请求在上游照样失败；我方只是让错误「干净、合规」。真实可用性 100% 取决于 B2 是否被上游修复。
  • 检测而非修复 模型降级嫌疑无法由我方消除。只能持续检测指纹、对比基线，决定保留 / 降权 / 下线该渠道。
  • 取决于评分口径 若评测 grader 只查「是否泄露 / 信封格式」→ 我方脱敏后可过；若还查「上游是否真能正确处理该请求」→ 仍 fail，且只能等上游 B2。
