A well-formed Anthropic Messages request fails inside your backend with an internal error instead of being processed. This is a functional failure — the request does not succeed, and there is no gateway-side workaround: a valid request that the upstream rejects cannot be made to succeed by us.

2. Proxy/validation layer converts via an OpenAI-format struct and breaks on standard fields P0 · ROOT CAUSE

代理/校验层用 OpenAI 结构体中转,在标准字段上失败(第 1 项的根因)

The internal error (GenerateOpenAIRequest) shows that a layer in front of the upstream model converts the inbound request into an OpenAI-format struct (likely for payment/validation), and that layer breaks on standard fields such as max_tokens. This is the mechanism behind item 1. (This is not a question of model authenticity — it is about the request-handling layer.)

ExpectedFix the OpenAI-conversion / validation layer so standard request fields (e.g. max_tokens) are handled correctly end-to-end and never surface an internal serializer error.
中文修复代理/校验层的 OpenAI 转换逻辑,使标准字段(如 max_tokens)端到端正确处理,不再抛出内部序列化错误。这是第 1 项的根因机制,与模型真伪无关。

4. Requests with sampling params are routed to a second backend that injects ~122 billed input tokens P0 · BILLING

带采样参数的请求被路由到第二后端，注入并计费 ~122 个 input token

Any request carrying temperature, top_p, or top_k (or context_management / an invalid thinking.enable) is routed to a different backend than clean requests — identifiable by a short message-id format — and that backend adds a fixed ~122 input tokens to every request and bills them. Because temperature is set by the vast majority of real clients, this inflates input-token billing across most production traffic.

Same prompt "hi": plain / valid thinking / stop_sequences → input_tokens=8 (long message id)
+ temperature | top_p | top_k | context_management → input_tokens=130 (short message id)
Longer prompt: plain=38 vs +temperature=160 → fixed delta = 122
Impact~122 phantom billed input tokens per request on most traffic. This second backend also skips parameter validation (item 5) and is very likely the same backend behind items 1–3 (internal-error leak, max_tokens failure, low model-quality scores).
ExpectedDo not inject prompt content that is billed to the customer; route all requests to a single consistent backend, or at minimum disclose and stop billing the injected tokens.
中文带采样参数的请求每条多计 ~122 个 input token，影响大部分真实流量；该后端还跳过参数校验，很可能就是第 1–3 项背后的同一后端。请勿对客户计费注入的 prompt 内容，并统一后端。
ReproduceSend the same prompt twice against your /v1/messages; the only difference is temperature. Compare usage.input_tokens and the id format.
BASE=<your /v1/messages endpoint>; KEY=<your key>
req(){ curl -s -X POST "$BASE" -H "x-api-key: $KEY" -H "anthropic-version: 2023-06-01" \
  -H "content-type: application/json" -d "$1" | grep -o '"id":"[^"]*"\|"input_tokens":[0-9]*'; }

# A) clean
req '{"model":"claude-opus-4-8","max_tokens":50,"messages":[{"role":"user","content":"hi"}]}'
# B) identical + temperature only
req '{"model":"claude-opus-4-8","max_tokens":50,"temperature":0.5,"messages":[{"role":"user","content":"hi"}]}'

// Observed via the channel: A → input_tokens=8, long id | B → input_tokens=130, short id
// Longer prompt: A=38 vs B=160 → fixed delta = 122
Pass / failPASS if B's input_tokens ≈ A's (no injection). FAIL if B ≈ A + ~122 and the id format differs — that confirms the second backend injects billed tokens. B 与 A 的 input_tokens 相近为通过；B ≈ A+122 且 id 格式不同即确认第二后端注入计费 token。

5. Parameter validation not enforced — invalid requests return 200 instead of 400 P1

参数校验未执行 — 非法请求返回 200 而非 400

Requests that the official Anthropic API rejects with 400 are accepted with 200:

  • top_p/temperature/top_k together with extended thinking (incompatible per Anthropic) → should be 400.
  • context_management sent without the required anthropic-beta header → should be 400.

Our gateway forwards these fields faithfully (verified by local round-trip) and does not inject the beta header, so the missing validation is upstream.

中文top_p/temperature/top_k 与 thinking 同用、context_management 缺 anthropic-beta 头，官方均应 400，实际 200。我方忠实转发且不注入 beta 头（本地已验证），校验缺失在上游。
