# 成本归因（P1）— 需求文档

> 状态：**已实现**（数据看板「成本归因」Tab）
> 所属：「AI 成本雷达」路线图的 **P1 阶段**（多维归因）。P0「完整请求/响应日志」已完成，见 `ai-cost-radar.md`。

---

## 1. 背景与定位

「AI 成本雷达」完整愿景分四期（见 `ai-cost-radar.md` §1.2）：

1. **P0 完整日志（已完成）**：落库每次请求/响应正文（含 prompt），媒体分离。
2. **P1 多维归因（本文档）**：把分散的调用成本，按 **用户 / 令牌 / 模型** 聚合、排序、可视化，回答"**谁/哪个 key/哪个模型在烧 token**"。
3. P2 浪费识别：标出"高成本低价值调用"（价值口径待定）。
4. P3 雷达页 + 优化建议引擎。

P1 是把现有"流水账日志"升级成"**可归因、可排序、能定位浪费点**的成本分析"。

### 关键事实（代码基线）
现有 `logs` 表已记录每次消费日志的结构化字段（`user_id` / `username` / `token_id` / `token_name` / `model_name` / `quota` / `prompt_tokens` / `completion_tokens` / `created_at` 等）。因此：

- **用户 / 令牌 / 模型 三维归因所需数据全部现成**，P1 **只做聚合查询 + 看板**，**不新增数据表、不依赖 P0 的 `request_details`**。
- 两个功能解耦：成本归因即使在 P0 日志开关关闭时也能正常工作（它读的是 `logs`，不是 `request_details`）。

---

## 2. 目标

- 管理员能在一个看板里，按 **时间范围 + 过滤条件**，看到成本（费用 / token / 调用次数）在 **用户、令牌、模型** 三个维度上的分布与排行。
- 能**下钻**：点开某一行（如某个令牌）查看它的**模型构成**占比（"这个 key 主要烧在哪些模型上"）。
- 能看**趋势**：成本随时间的变化。

---

## 3. 范围

### 3.1 范围内（In Scope）

- **放置位置**：在「数据看板」页（`web/default` 的 `features/dashboard`）新增一个 **「成本归因」Tab**，与"概览 / 模型调用分析 / 用户统计"并列（仅管理员可见）。归因属于"分析"语义，与看板既有的用户/模型排行+趋势同源，放在一起最连贯；而看板的用户统计/模型分析走预聚合 `quota_data` 表（无令牌粒度），成本归因则直查 `logs` 表，补齐了 **令牌维度 + 组合下钻** 的缺口。
- **时间与范围控制**（复用看板约定）：时间范围预设（7/30/... 天滚动）+ Top-N 选择器。不复用日志明细那条细粒度文本筛选栏（看板定位为宏观分析）。
- **汇总卡片**：总费用、总输入 token、总输出 token、总调用次数。
- **三维排行**：维度选择器（用户 / 令牌 / 模型），展示该维度的排行表——列：名称、费用、输入 token、输出 token、调用次数、费用占比；默认按费用倒序。
- **二级下钻（组合归因）**：点排行表任意一行 → 展开其下一层构成：
  - 令牌行 → **令牌 × 模型**（核心诉求：该 key 在各模型上的占比）
  - 用户行 → 用户 × 模型
  - 模型行 → 模型 × 令牌（该模型主要被哪些 key 使用）
- **趋势折线**：所选维度 Top-N 的费用随时间（按天）变化。
- **鉴权**：所有聚合接口在服务端强制 `middleware.AdminAuth()`，仅管理员可见。

### 3.2 范围外（Out of Scope）

- **prompt 模板归因**（模板指纹聚类）—— 本期明确不做。
- "功能"维度 —— 用**令牌（Token）**替代（团队按 key 区分应用，更易区分），不引入独立的"功能"概念。
- "高成本低价值"打分与价值口径（P2）。
- 替代模型 / 缓存 / 压缩 prompt 等优化建议（P3）。
- 独立的成本雷达可视化大页（P3）。
- 预聚合表 / 物化视图等性能优化（P1 先直查 `logs`，数据量大时再议）。

### 3.3 已确认的口径

| 决策点 | 结论 |
|--------|------|
| "功能"维度 | **用令牌（Token）替代**，不叫"功能"（不好区分） |
| prompt 模板 | **本期不做** |
| 组合下钻 | **做**，支持 令牌×模型 / 用户×模型 / 模型×令牌 |
| 看板位置 | **数据看板页新增「成本归因」Tab**（与 概览/模型调用分析/用户统计 并列，仅管理员） |
| 数据来源 | 仅 `logs` 表（`type = LogTypeConsume`），不依赖 `request_details` |
| 可见范围 | 仅管理员（服务端 `AdminAuth()` 强制） |

## 4. 指标定义

| 指标 | 口径 | 来源字段 |
|------|------|----------|
| 费用（Cost） | `SUM(quota)`，前端按 `quota / QuotaPerUnit` 折算为金额展示（与现有日志页一致） | `logs.quota` |
| 输入 token | `SUM(prompt_tokens)` | `logs.prompt_tokens` |
| 输出 token | `SUM(completion_tokens)` | `logs.completion_tokens` |
| 总 token | 输入 + 输出 | 同上 |
| 调用次数 | `COUNT(*)` | — |
| 平均每次费用 | 费用 / 调用次数 | — |
| 占比 | 该行费用 / 当前结果集总费用 | — |

> 统计范围统一限定 `type = LogTypeConsume`（消费日志），与 `SumUsedQuota` 保持一致；错误日志（`LogTypeError`，quota=0）不计入费用，但其调用次数是否纳入"次数"统计在实现时确认（默认只统计消费日志）。

## 5. 交互与展示

**Tab 切换**：数据看板页顶部 `概览 | 模型调用分析 | 用户统计 | 成本归因`（成本归因仅管理员可见），由 URL `section` 路由参数驱动。

**成本归因 Tab 布局（自上而下）**：
1. 控制栏：维度选择器（用户 / 令牌 / 模型）+ 时间范围预设 + Top-N 选择器。
2. 汇总卡片行（总费用、调用次数、输入/输出 token）。
3. 每日费用趋势（VChart 面积图，所选维度 Top-N 系列；与用户统计/模型分析视觉一致）。
4. 排行表：可展开行（令牌/用户行点击展开 ×模型 构成，带占比条）。

**展示细节**：
- 趋势图复用看板 VChart 主题与样式；金额经 `formatLogQuota` 本地化展示。
- 空数据、加载中态友好提示。
- 全部文案走 i18n（`useTranslation`，6 语言）。

- [ ] 数据看板页出现「成本归因」Tab（仅管理员），与其他看板 Tab 切换流畅。
- [ ] 汇总卡片数值与同条件下日志明细的合计一致（费用、token、次数）。
- [ ] 三个维度（用户/令牌/模型）排行正确，按费用倒序，占比合计 ≈ 100%。
- [ ] 下钻：点令牌行能看到该令牌的 **模型构成** 及占比；用户行、模型行同理。
- [ ] 趋势折线按天正确反映所选维度 Top-N 的费用变化。
- [ ] 时间范围 / 用户 / 令牌 / 模型 / 分组等过滤条件对聚合与下钻均生效。
- [ ] **接口鉴权**：用非管理员 token 直接调用聚合接口返回无权限（403），证明权限在服务端生效。
- [ ] **三库兼容**：SQLite / MySQL / PostgreSQL 下聚合、下钻、趋势查询均正确（重点验证按天分桶与 `COALESCE`）。
- [ ] 与 P0 解耦：P0 日志开关关闭时，成本归因仍可正常使用。

## 7. 风险与缓解

| 风险 | 说明 | 缓解 |
|------|------|------|
| 大数据量聚合慢 | `logs` 海量时 `GROUP BY` 扫描成本高 | 复用现有 `created_at` / `user_id` / `model_name` / `token_id` 索引；限定时间范围；Top-N 限制返回行数；必要时 P1+ 引入按天预聚合表 |
| 跨库聚合差异 | `ifnull` / 整除 / 保留字列差异 | 统一用 `COALESCE`；按天分桶用 `DIV`(MySQL)/`/`(PG/SQLite) 分支；`group` 列用 `logGroupCol` |
| 令牌/用户被删改 | `token_name` 是快照、可能重名或被删 | 以 `token_id` / `user_id` 为聚合主键，`*_name` 作展示标签 |
| 时区 | 按天分桶受时区影响 | 明确按服务器/UTC 时间分桶，前端标注口径（实现时确认是否需按用户时区偏移）|
