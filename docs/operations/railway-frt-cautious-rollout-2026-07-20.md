## Codex 上游 429 中断事故

- 客户端现象：`exceeded retry limit, last status: 429 Too Many Requests`。
- 事故时间：2026-07-20 17:10:54-17:10:57（Asia/Singapore）。
- 网关 request ID：`202607200910541964497948268d9d6jKaFaiX2`。
- 请求路径：`POST /v1/responses`，客户端为 Codex Desktop。
- 失败链路：渠道 `#68 -> #59 -> #70` 依次返回 `Too many pending requests` 或
  `Upstream rate limit exceeded`，配置内重试全部耗尽后向客户端返回 HTTP 429。
- 17:43 的生产部署 `cc54c60f-b358-42c1-ba5f-245dbaef2443` 之后仍持续观察到多个渠道的
  同类 429，说明这是上游账户池容量波动，不是单个渠道或单次请求异常。

第一轮修复只扩大可安全恢复的边界，不在请求中排队或睡眠：

- 收到上游 HTTP 状态前不再向下游写 ping、SSE header 或隐式提交 HTTP 200；延迟返回的
  HTTP 429 因而仍可在响应未提交时安全切换渠道。
- 真实上游 429 在配置内重试耗尽后，最多额外尝试一个尚未使用的渠道；auto group 会重置
  选择游标，但保留已试渠道集合，避免重复选择和无限重试。
- 已写下游响应、已收到流数据、指定渠道、skip-retry、语义错误、账户余额/配额不足以及
  affinity 禁止切换的请求均不进入额外尝试。
- 额外尝试没有可用渠道时保留原始上游 429；预扣费仍只执行一次，成功尝试才结算，不增加
  重复扣费路径。
- 已经以 HTTP 200 开始后才出现的 SSE `Concurrency limit exceeded for account` 属于另一条
  协议边界，不能透明重放；继续依靠失败流识别、affinity 抑制和渠道冷却处理。

修复提交为 `5524d6e40`，并已通过全量 Go 测试、定向 race 测试、Go vet、前端生产构建和
独立代码审查。上线前又合入当前生产 `a566864ac` 的定价改动，合并提交为 `62f41f401`，避免
部署时回退线上功能。

第二轮谨慎恢复提交为 `52e1fb5b7`，边界进一步收紧：

- 容量错误只接受两类：排除余额/配额错误后的真实上游 429，以及精确匹配

- 触发恢复的容量错误本身必须在 2 秒内返回；配置内重试耗尽后最多再选择两个未尝试渠道，

- 预扣费仍在重试循环外执行一次；成功只结算一次，最终失败只退款一次。已经提交 HTTP 200

## 保留第二个备用渠道机会

生产证据表明，两次额外尝试虽然共享 5 秒绝对窗口，但首个额外渠道可以独占整个窗口，
使第二个未尝试渠道没有执行机会。第三轮调整不增加最大尝试数，也不增加总等待：

- 首个额外渠道最多等待 2 秒完整响应头；若内部响应头切片到期且总 5 秒窗口仍有余量，立即
  选择第二个未尝试渠道；
- 第二个额外渠道使用 5 秒绝对窗口的剩余时间，绝不重新开始一个 5 秒计时；
- 内部切片到期发生在渠道健康、冷却和共享主机归因之前，不把谨慎切换误记为渠道故障；
- 客户端取消仍优先返回 `context.Canceled`，已提交响应、已有流数据、非流式请求、普通
  502/503、配额/余额错误和长 SSE body 均不进入该切片逻辑；
- 若第二个渠道也不可用或总窗口耗尽，继续保留最后一个真实上游容量错误，不向客户端泄漏
  内部 deadline sentinel。

该调整已通过全量 Go 测试、定向 race 测试、Go vet、`git diff --check` 和静默失败复核。
上线后重点观察 `upstream capacity fallback response-header slice reached`、
`capacity fallback succeeded` 以及相同请求的渠道链路和总耗时。

## 上游 429 两小时冷却与切换

上线前基线仍为 deployment `4dc97f88-366f-43c3-9834-0a458925ceca`。截至 2026-07-20
21:20（Asia/Singapore），该版本累计观察到 394 个 `/v1/responses` 请求：382 个 HTTP 200、
4 个 500、4 个 502、4 个 503，最终 HTTP 429 为 0。精确检索 `Too Many Requests`、
`Concurrency limit exceeded`、`Upstream rate limit exceeded`、`upstream_status=429` 和
`status code: 429` 均无结果，因此当前没有可用于验证两小时冷却的自然 429 样本。现网仅有余额
不足的 30 分钟冷却和普通 500/502/503 的 5 分钟冷却，没有两小时冷却记录。

本轮策略只识别 `UpstreamStatusCode == 429` 的真实上游限流，避免把网关自身的 429 归因到渠道：

- 任何真实上游 429，包括客户端状态已映射为 400/503 或消息为 `insufficient_quota` 的情况，
  都将该渠道冷却 2 小时；后续较短冷却不能缩短已有期限，新的 429 会从最新时间重新延长 2 小时；
- 当前请求已在发起上游调用前把渠道 ID 加入 `ExcludedChannelIDs`，因此切换时不会再次选择刚刚
  返回 429 的渠道；后续请求优先使用非冷却渠道，只有全部正常候选耗尽时才把冷却渠道作为最后
  兜底，保持“尽量避开”而不是重新制造无可用渠道 503；
- 不增加配置内重试次数、额外恢复次数或 5 秒容量恢复窗口；固定渠道、渠道亲和性、非流式/副作用
  请求和已提交响应继续保留既有限制；`insufficient_quota` 等结构性配额 429 只使用配置内重试，
  不进入额外两次容量恢复；
- 真实上游 429 不再触发永久自动禁用，即使运营侧把 429 加入自动禁用状态范围；任务提交路径也
  保留原始上游状态，使用相同冷却策略；普通任务可以切换到未尝试渠道，必须绑定原始渠道的任务
  遇到 429 后立即停止，避免对同一锁定来源做无效重复提交；AWS Bedrock SDK 只有在取得真实 HTTP
  响应状态时才记录 upstream provenance，本地客户端错误不会伪装成上游 429；
- 冷却状态沿用现有按渠道 ID 的进程内存表：当前单副本部署可直接生效，但容器重启会清空，未来
  若扩容为多副本需迁移到共享状态；多 Key 渠道仍按整个渠道冷却，这是本轮明确采用的保守渠道级
  策略，避免引入只有部分选择路径识别 Key 冷却的负向优化。

## 首字慢渠道跨优先级软降级

生产基线确认：现有 EWMA 和快渠道权重只会在同一优先级内调整，无法处理“高优先级但首字持续慢”
的渠道。此前窗口中，#42 `aiccxx.top`（plus，优先级 11）和 #79 `strategyhub.cc`
（pro，优先级 12）均有稳定的 8-16 秒首字样本，却始终压在更快的低一层渠道之前。

本轮采用单层、可恢复的软降级：

- 同一 `(channel, model, normalized path)` 最近两个可归因、非冷启动的成功首字达到慢阈值（默认
  9 秒）后，只把有效优先级下移一层；不写入渠道冷却，不改变 affinity 粘滞路由，也不把冷缓存
  prefill 当成渠道慢。
- 继续出现第三个慢样本时，沿用现有慢熔断和 half-open 探测；慢熔断恢复不会清掉软降级标记，
  必须有两个连续可计分的快响应且 EWMA 低于慢阈值才恢复原层级。
- 失败、慢响应会重新计时；冷启动、零首字、客户端/语义/本地错误只清掉连续快样本，不会把探测
  窗口无限推迟。恢复探测机会只在实际加权选中渠道后原子预留，缓存和数据库选路共用同一规则；
  过期候选快照仍经过普通健康/熔断 Acquire，不能绕过已打开的 circuit。
- 最低配置优先级不再额外下移；没有可用的高层候选时，仍保留原有 fallback 和探测机会。
