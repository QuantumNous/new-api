## Codex 上游 429 中断事故

- 客户端现象：`exceeded retry limit, last status: 429 Too Many Requests`。
- 事故时间：2026-07-20 17:10:54-17:10:57（Asia/Singapore）。
- 网关 request ID：`202607200910541964497948268d9d6jKaFaiX2`。
- 请求路径：`POST /v1/responses`，客户端为 Codex Desktop。
- 失败链路：渠道 `#68 -> #59 -> #70` 依次返回 `Too many pending requests` 或
  `Upstream rate limit exceeded`，配置内重试全部耗尽后向客户端返回 HTTP 429。
- 17:43 的生产部署 `cc54c60f-b358-42c1-ba5f-245dbaef2443` 之后仍持续观察到多个渠道的
  同类 429，说明这是上游账户池容量波动，不是单个渠道或单次请求异常。

第一轮修复只扩大可安全恢复的边界，不在请求中排队或睡眠：

- 收到上游 HTTP 状态前不再向下游写 ping、SSE header 或隐式提交 HTTP 200；延迟返回的
  HTTP 429 因而仍可在响应未提交时安全切换渠道。
- 真实上游 429 在配置内重试耗尽后，最多额外尝试一个尚未使用的渠道；auto group 会重置
  选择游标，但保留已试渠道集合，避免重复选择和无限重试。
- 已写下游响应、已收到流数据、指定渠道、skip-retry、语义错误、账户余额/配额不足以及
  affinity 禁止切换的请求均不进入额外尝试。
- 额外尝试没有可用渠道时保留原始上游 429；预扣费仍只执行一次，成功尝试才结算，不增加
  重复扣费路径。
- 已经以 HTTP 200 开始后才出现的 SSE `Concurrency limit exceeded for account` 属于另一条
  协议边界，不能透明重放；继续依靠失败流识别、affinity 抑制和渠道冷却处理。

修复提交为 `5524d6e40`，并已通过全量 Go 测试、定向 race 测试、Go vet、前端生产构建和
独立代码审查。上线前又合入当前生产 `a566864ac` 的定价改动，合并提交为 `62f41f401`，避免
部署时回退线上功能。

第二轮谨慎恢复提交为 `52e1fb5b7`，边界进一步收紧：

- 容量错误只接受两类：排除余额/配额错误后的真实上游 429，以及精确匹配

- 触发恢复的容量错误本身必须在 2 秒内返回；配置内重试耗尽后最多再选择两个未尝试渠道，

- 预扣费仍在重试循环外执行一次；成功只结算一次，最终失败只退款一次。已经提交 HTTP 200

## 保留第二个备用渠道机会

生产证据表明，两次额外尝试虽然共享 5 秒绝对窗口，但首个额外渠道可以独占整个窗口，
使第二个未尝试渠道没有执行机会。第三轮调整不增加最大尝试数，也不增加总等待：

- 首个额外渠道最多等待 2 秒完整响应头；若内部响应头切片到期且总 5 秒窗口仍有余量，立即
  选择第二个未尝试渠道；
- 第二个额外渠道使用 5 秒绝对窗口的剩余时间，绝不重新开始一个 5 秒计时；
- 内部切片到期发生在渠道健康、冷却和共享主机归因之前，不把谨慎切换误记为渠道故障；
- 客户端取消仍优先返回 `context.Canceled`，已提交响应、已有流数据、非流式请求、普通
  502/503、配额/余额错误和长 SSE body 均不进入该切片逻辑；
- 若第二个渠道也不可用或总窗口耗尽，继续保留最后一个真实上游容量错误，不向客户端泄漏
  内部 deadline sentinel。

该调整已通过全量 Go 测试、定向 race 测试、Go vet、`git diff --check` 和静默失败复核。
上线后重点观察 `upstream capacity fallback response-header slice reached`、
`capacity fallback succeeded` 以及相同请求的渠道链路和总耗时。
