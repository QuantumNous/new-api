# Roadmap

本文档记录当前项目的阶段性规划，作为后续开发、发布、部署与功能演进的执行参考。

## 1. 开发流水线

首先搭建完整的开发流水线，覆盖本地可运行的基础验证能力与持续集成发布能力。

当前状态（2026-03-21）：

- 已落地统一入口 `make verify`
- 已覆盖单元测试、API 测试、E2E 测试、升级兼容性测试
- 已有 `.github/workflows/verify.yml` 在 `pull_request` 与普通 `push` 自动执行仓库级验证
- 当前主线优先级可切换到第 2 项“升级流水线”

### 本地验证能力

- 单元测试
- API 测试
- E2E 测试
- 升级兼容性测试

目标：

- 可以在本机完成主要验证流程
- 在提交前尽早发现回归问题
- 为后续自动化 CI 打好基础

### CI / 发布流水线

- 建立 CI 流程，自动执行单元测试等验证步骤
- 建立版本发布流程
- 自动更新并发布 Docker 镜像

目标：

- 降低手工发布成本
- 提高版本发布的一致性与可追踪性

## 2. 升级流水线

升级流程分阶段推进，优先落地两种可用方式，再逐步演进为更完善的发布体系。

### 方式一：服务器 SSH 终端手动升级

- 在服务器的 SSH terminal 中执行脚本
- 手动升级到指定版本
- 尽量减少服务 down time

### 方式二：前端管理员页面触发升级

- 在前端管理员页面点击升级
- 先触发后台升级流程
- 再更新前端资源
- 页面通过手动刷新完成前端更新

### 后续演进

- 引入更合理的蓝绿部署等能力
- 逐步提升升级过程的稳定性、可回滚性与可观测性

## 3. 分布式部署

围绕生产环境的可用性与扩展性，逐步支持更清晰的分布式部署方式。

### 基础目标

- DB、Redis 与应用服务分离部署
- 服务节点与数据节点职责拆分

### 高可用目标

- 主从部署
- 高可用能力建设

## 4. 功能增强

功能增强可以穿插在任意阶段推进，不必严格依赖主线里程碑完成后再开始。

### 视觉与品牌调整

- 换网站 logo
- 换网站名字

备注：

- 涉及项目受保护标识、名称、品牌信息时，仍需遵循仓库内既有项目策略与约束

### 支付方式扩展

- 新的支付方式
- 例如 USDT / USDC
- 例如 ERC20、TRC20 等

### 监控与可观测性增强

- 大盘监控
- 渠道可用性看板
- API 延迟监控
- 首 token 延迟监控
- 面向用户侧和管理员侧的监控视图

## 建议的执行顺序

建议优先顺序如下：

1. 开发流水线（已完成当前基线）
2. 升级流水线（当前下一步）
3. 分布式部署
4. 功能增强按业务价值穿插推进
