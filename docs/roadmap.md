# Roadmap

本文档记录当前项目的阶段性规划，作为后续开发、发布、部署与功能演进的执行参考。

## 1. 开发流水线

首先搭建完整的开发流水线，覆盖本地可运行的基础验证能力与持续集成发布能力。

当前状态（2026-03-21）：

- 已落地统一入口 `make verify`
- 已覆盖单元测试、API 测试、E2E 测试、升级兼容性测试
- 已有 `.github/workflows/verify.yml` 在 `pull_request` 与普通 `push` 自动执行仓库级验证
- 已打通 tag 驱动的 Docker 镜像发布链路，支持推送多架构镜像到已配置的 Docker Hub 仓库
- Docker 发布 workflow 已改为显式校验 `DOCKERHUB_REPOSITORY`、`DOCKERHUB_USERNAME`、`DOCKERHUB_TOKEN`
- fork 当前版本基线已重置为 `v1.0.0`，后续版本可独立迭代
- 当前主线优先级可切换到第 2 项“升级流水线”

### 本地验证能力

- 单元测试
- API 测试
- E2E 测试
- 升级兼容性测试

目标：

- 可以在本机完成主要验证流程
- 在提交前尽早发现回归问题
- 为后续自动化 CI 打好基础

### CI / 发布流水线

- 建立 CI 流程，自动执行单元测试等验证步骤
- 建立版本发布流程
- 自动更新并发布 Docker 镜像

当前补充说明：

- 正式镜像发布采用 tag 驱动
- 预发布测试 tag 可验证镜像推送链路且不会覆盖 `latest`
- 镜像仓库不再写死在 workflow 中，统一由 GitHub Actions variable / secret 提供

目标：

- 降低手工发布成本
- 提高版本发布的一致性与可追踪性

## 2. 升级流水线

升级流程分阶段推进，优先落地两种可用方式，再逐步演进为更完善的发布体系。

当前状态（2026-03-21）：

- 升级兼容性测试已纳入 `make verify`
- 仓库已有“升级验证”基础，但“面向生产环境的升级执行链路”仍未成型
- 当前该章节是主线下一步，需要从“验证升级可兼容”推进到“系统可被稳定升级”

### 阶段拆分

建议按以下顺序推进：

1. 先落地服务器 SSH 手动升级方案，确保线上可控升级
2. 再补管理员页面触发升级，降低运维门槛
3. 最后补回滚、灰度、蓝绿部署与观测能力

### 阶段 2.1：服务器 SSH 手动升级

目标：

- 在单机或单节点部署下，可以通过标准脚本完成一次完整升级
- 升级过程可重复执行，失败时可快速定位
- 升级过程尽量减少停机时间，并保留回滚入口

建议产物：

- `scripts/release/` 下的升级脚本或入口脚本
- 版本包获取逻辑（git tag、release artifact 或 Docker 镜像三选一，避免多套并行）
- 升级前检查：磁盘空间、配置文件、数据库连接、Redis 连通性
- 升级执行步骤：拉取版本、备份关键文件、执行数据库迁移、重启服务、健康检查
- 升级失败后的回滚说明或回滚脚本
- 管理员可直接执行的运维文档

验收标准：

- 能从一个已发布版本升级到下一个已发布版本
- 能在升级后完成基础健康检查与关键 API 冒烟验证
- 能明确区分“前置检查失败”“迁移失败”“服务启动失败”“健康检查失败”
- 至少覆盖一条升级兼容性验证路径，并与现有 `make verify` 基线衔接

### 方式一：服务器 SSH 终端手动升级

- 在服务器的 SSH terminal 中执行脚本
- 手动升级到指定版本
- 尽量减少服务 down time

### 阶段 2.2：前端管理员页面触发升级

目标：

- 管理员无需登录服务器，也能触发一次受控升级
- 前端只负责发起与展示状态，真正的升级逻辑由后端统一调度
- 整个过程需要有权限控制、状态记录与失败可见性

### 方式二：前端管理员页面触发升级

- 在前端管理员页面点击升级
- 先触发后台升级流程
- 再更新前端资源
- 页面通过手动刷新完成前端更新

建议产物：

- 后端升级任务接口：创建任务、查询状态、查询日志
- 任务状态机：待执行、执行中、成功、失败、已回滚
- 后台任务日志持久化或临时缓存方案
- 管理员页面：触发升级、查看进度、查看结果、查看失败原因
- 权限与审计：仅管理员可操作，保留操作人和触发时间

验收标准：

- 页面可查看升级任务实时状态
- 同一时间只允许一个升级任务执行，避免并发破坏部署
- 升级失败时前端可看到明确错误，而不是仅显示超时或未知错误
- 管理端升级流程不绕过现有发布与版本来源策略

### 阶段 2.3：升级安全与回滚增强

目标：

- 将升级从“可执行”推进到“可恢复、可观测、可审计”

建议范围：

- 升级前自动备份配置、静态资源与关键数据库快照
- 支持应用版本回滚
- 健康检查失败时自动停止切流或自动回退
- 记录版本来源、升级人、升级时间、升级耗时
- 建立升级成功率、失败率、平均耗时等指标

### 后续演进

- 引入更合理的蓝绿部署等能力
- 逐步提升升级过程的稳定性、可回滚性与可观测性

## 3. 分布式部署

围绕生产环境的可用性与扩展性，逐步支持更清晰的分布式部署方式。

### 基础目标

- DB、Redis 与应用服务分离部署
- 服务节点与数据节点职责拆分

建议下一步：

- 先明确单节点部署与多节点部署的配置差异
- 抽离会话、限流、缓存等对本地状态的依赖
- 校验后台任务、升级任务与定时任务在多节点下的归属和互斥机制

### 高可用目标

- 主从部署
- 高可用能力建设

建议验收方向：

- 单节点故障不会导致数据库或缓存层同时失效
- 多节点部署下，关键后台任务不会重复执行
- 升级流程在多节点场景中可实现分批或滚动升级

## 4. 功能增强

功能增强可以穿插在任意阶段推进，不必严格依赖主线里程碑完成后再开始。

### 视觉与品牌调整

- 换网站 logo
- 换网站名字

备注：

- 涉及项目受保护标识、名称、品牌信息时，仍需遵循仓库内既有项目策略与约束

### 支付方式扩展

- 新的支付方式
- 例如 USDT / USDC
- 例如 ERC20、TRC20 等

建议前置条件：

- 先统一支付抽象与回调状态流转
- 明确链上支付的汇率、确认数、风控与补单逻辑
- 补齐管理员侧对账、异常订单处理与审计入口

### 监控与可观测性增强

- 大盘监控
- 渠道可用性看板
- API 延迟监控
- 首 token 延迟监控
- 面向用户侧和管理员侧的监控视图

建议拆分：

- 第一阶段先补服务健康、错误率、延迟、任务状态等基础指标
- 第二阶段补渠道维度、模型维度、租户维度监控
- 第三阶段再做用户侧可见的大盘和告警联动

## 交付方式约束

Roadmap 中涉及的开发、发布、升级与主线更新，统一遵循以下方式：

- 所有面向 `main` 的更新通过 feature branch + Pull Request 合入
- 依赖 CI 校验通过后，再由代码托管平台执行 merge
- 不采用本地直接合并 `main` 后再 `push` 的方式推进主线

该约束用于保证升级、发布与 roadmap 执行过程具备可审查性与可追踪性。

## 建议的执行顺序

建议优先顺序如下：

1. 开发流水线（已完成当前基线）
2. 升级流水线阶段 2.1：服务器 SSH 手动升级
3. 升级流水线阶段 2.2：管理员页面触发升级
4. 升级流水线阶段 2.3：回滚与可观测性增强
5. 分布式部署
6. 功能增强按业务价值穿插推进
