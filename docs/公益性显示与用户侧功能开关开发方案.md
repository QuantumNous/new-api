# 公益性显示与用户侧功能开关开发方案

## 背景

2026-05-24 的修改把“钱包、充值、额度”等中文翻译直接改成了当前站点想要的“支持中心、项目支持、点数”，同时用前端硬编码方式隐藏了一部分用户侧入口。这个做法能快速解决当前站点展示问题，但不适合继续作为正式开发方向。

核心问题有三个：

- 站点私有文案污染了默认中文翻译，上游或其他部署方会被迫看到同一套公益性文案。
- 部分显示逻辑是硬隐藏，不是配置驱动，后续想开关邀请、充值、订阅、兑换码时会继续出现零散补丁。
- 新版前端和旧版前端并存，只改一套或只改翻译文件，会出现入口名称、卡片显示、接口请求不一致。

本方案目标不是继续硬改页面，而是把需求拆成两个独立能力：

- **用户侧功能开关**：控制钱包充值、订阅购买、兑换码使用、邀请面板等入口是否显示和是否允许调用。
- **公益性显示模式**：控制用户侧文案是否从“钱包/充值/余额/额度”切换为“支持中心/项目支持/可用点数/模型点数”等更中性的展示方式。

## 需求拆分

### 已经具备的能力

现有代码已经有一套支付业务能力和支付场景解耦：

- `payment_setting.business_features.wallet_topup` 控制用户钱包充值。
- `payment_setting.business_features.subscription_purchase` 控制用户订阅购买。
- `payment_setting.business_features.redemption_redeem` 控制用户兑换码使用。
- `payment_setting.provider_scene_scopes` 控制 `epay`、`stripe`、`creem`、`waffo`、`waffo_pancake` 是否可用于 `wallet_topup` 或 `subscription_purchase`。

相关代码已经覆盖后端 guard、新旧前端支付设置页、钱包/订阅页的主要链路。

### 仍缺少的能力

当前还缺少这些正式化设计：

- 邀请面板没有独立显示开关。旧版 `InvitationCard` 固定渲染，并且固定请求 `/api/user/aff`；新版源码里有 `AffiliateRewardsCard` 和 `useAffiliate`，但当前钱包页没有实际挂载。
- “公益性显示”没有配置，只是直接改 `web/default/src/i18n/locales/zh.json`，旧版页面里的中文仍大量保留“钱包、充值、额度”。
- `wallet_topup=false` 后，用户侧应该隐藏对应区域，而不是显示“未开启充值”的提示卡，尤其在订阅仍可购买时不应该误导用户。
- 旧版用户菜单和侧边栏里还有“钱包管理”入口文字，不能只改钱包页内部。
- 如果关闭邀请面板，前端也应避免无意义请求 `/api/user/aff` 和 `/api/user/aff_transfer`。

## 推荐配置设计

### 保持支付能力边界不变

继续使用现有 `payment_setting.business_features`，只表达“用户侧支付/兑换入口”：

```json
{
  "wallet_topup": true,
  "subscription_purchase": true,
  "redemption_redeem": true
}
```

不建议重新加入：

- `invitation_reward`
- `invitation_transfer`
- `checkin_reward`
- `redemption_manage`

原因是这些键已经在 2026-05-20 的设计中被收敛掉，重新加入会让“业务能力”和“已有独立配置/管理入口”重复控制，产生上下设计冲突。

### 新增用户侧显示配置

新增独立配置模块：

```go
type BillingDisplaySetting struct {
    PublicWelfareTextEnabled bool `json:"public_welfare_text_enabled"`
    InvitationPanelEnabled   bool `json:"invitation_panel_enabled"`
}
```

注册名：

```text
billing_display_setting
```

默认值：

```json
{
  "public_welfare_text_enabled": false,
  "invitation_panel_enabled": true
}
```

默认值必须保持上游行为：

- 默认不启用公益性文案，仍显示“Wallet / 钱包 / 充值 / 额度”等原有文案。
- 默认显示邀请面板，避免升级后已有站点的邀请功能突然消失。

### 为什么不用 `payment_setting`

不建议把这两个字段放进 `payment_setting`，原因：

- 公益性显示是纯 UI/文案策略，不是支付能力。
- 邀请面板是用户增长/展示策略，不等同于支付通道能力。
- `payment_setting` 已经承担支付能力、支付场景、合规确认、金额挡位，继续塞显示开关会让配置语义变浑。

`billing_display_setting` 更符合实际含义：它控制用户侧账务/支持相关展示，而不是控制账务计算本身。

## 后端接口设计

### `/api/status`

在 `controller/misc.go:GetStatus` 返回：

```json
{
  "billing_display": {
    "public_welfare_text_enabled": false,
    "invitation_panel_enabled": true
  }
}
```

用途：

- 侧边栏、用户下拉菜单、页面标题等全局入口可以在不请求 `/api/user/topup/info` 的情况下拿到显示模式。
- 新版 `system-config-store` 可以持久化该配置，供全局 hooks 使用。
- 旧版 `StatusContext` 和 `localStorage.status` 可以直接读取。

### `/api/user/topup/info`

在 `GetTopUpInfo` 里同步返回：

```json
{
  "display": {
    "public_welfare_text_enabled": false,
    "invitation_panel_enabled": true
  }
}
```

用途：

- 钱包页内部不必额外依赖全局 status 的加载时序。
- `RechargeCard`、`SubscriptionPlansCard`、`InvitationCard` 可以用同一份 topup info 判断显示和文案。

### 邀请接口的后端边界

第一期只做“显示和前端请求开关”，不建议直接禁止 `/api/user/aff`。理由：

- 邀请链接可能仍被注册流程使用。
- 当前需求是“不需要邀请面板”，不是废除邀请码系统。
- 后端奖励发放已经由 `QuotaForInviter`、`QuotaForInvitee` 和合规确认约束。

如果后续要严格禁用邀请系统，可以新增更明确的配置：

```json
{
  "invitation_registration_enabled": true,
  "invitation_reward_enabled": true,
  "invitation_transfer_enabled": true
}
```

但这不是本次最小正确范围。

### 新版管理端

在 `payment-settings-section.tsx` 中新增一个“用户侧显示”配置块，放在“业务能力开关”附近，但视觉上分区：

- 公益性显示模式
- 显示邀请面板

- `billing_display_setting.public_welfare_text_enabled`

### 旧版管理端

在 `SettingsGeneralPayment.jsx` 中同样新增“用户侧显示”配置块。

旧版可复用 Semi `Switch`，和现有业务能力开关风格一致。

保存时把变更加入 `options`：

```js
{
  key: 'billing_display_setting.public_welfare_text_enabled',
  value: String(inputs.PublicWelfareTextEnabled)

```

```js
{

  value: String(inputs.InvitationPanelEnabled)

```

### 是否放在支付设置里

可以放在支付设置页，但标题要避免误导。推荐：

```text

```

```text
仅影响用户侧文案和入口展示，不改变模型计费、余额计算、订单和订阅扣费逻辑。
```

这样既方便运营配置，也不会让人误以为它改变了真实计费。

## 公益性文案模式设计

### 设计原则

- 默认模式完全保留原文案。
- 公益性模式只影响用户侧可见文案，不影响数据库字段、API 字段、日志类型、模型计费、quota 内部单位。
- 不再直接改 `zh.json` 里通用词条的默认翻译。
- 新版和旧版都通过一个文案 helper 获取显示文本。
- 管理端技术配置仍保留准确术语，如 `quota`、`billing`、`payment`，避免管理员误解配置含义。

### 推荐词表

公益性模式下用户侧词表建议：

| 原概念 | 默认显示 | 公益性显示 |
| --- | --- | --- |
| Wallet | 钱包/钱包管理 | 支持中心 |
| Add Funds / Recharge | 充值/账户充值 | 项目支持 |
| Balance | 余额 | 可用点数 |
| Quota | 额度 | 模型点数 |
| Total Quota | 总额度 | 总点数 |
| Raw Quota | 原生额度 | 系统原生点数 |
| Billing History / Order History | 账单/订单历史 | 支持记录 |
| Transfer to Balance | 划转到余额 | 转入可用点数 |
| Wallet First | 优先钱包 | 优先可用点数 |
| Wallet Only | 仅用钱包 | 仅用可用点数 |
| Top-up Amount | 充值金额 | 支持金额 |

### 新版实现方式

新增 helper：

```text
web/default/src/features/wallet/lib/display-text.ts
```

或更通用：

```text
web/default/src/lib/billing-display.ts
```

推荐暴露：

```ts
function getBillingDisplayText(
  key: BillingDisplayTextKey,
  t: TFunction,
  publicWelfare: boolean
): string
```

也可以提供 hook：

```ts
function useBillingDisplayText()
```

使用范围：

- 钱包页标题和描述。
- 侧边栏 `Wallet`。
- 用户下拉菜单 `Wallet Management`。
- 充值卡片标题、按钮、输入 label。
- 订阅偏好中的 `Wallet First`、`Wallet Only`。
- 订单历史弹窗标题。
- 邀请收益转入按钮。
- 统计卡中的余额/已消耗/总额度。

### 旧版实现方式

新增 helper：

```text
web/classic/src/helpers/billingDisplay.js
```

推荐暴露：

```js
export function getBillingDisplayText(key, t, publicWelfare)
```

旧版使用范围：

- `SiderBar.jsx` 的 `钱包管理`。
- `UserArea.jsx` 的 `钱包管理`。
- `RechargeCard.jsx` 的 `账户充值`、`额度充值`、`充值数量`、`选择充值额度`、`兑换码充值`。
- `SubscriptionPlansCard.jsx` 的 `优先钱包`、`仅用钱包`、`总额度`、`原生额度`。
- `TransferModal.jsx` 的 `划转邀请额度`、`可用邀请额度`、`划转额度`。
- `InvitationCard.jsx` 的收益统计和转入按钮。

### 不应修改的范围

以下地方不建议跟随公益性显示模式修改：

- 管理端模型倍率、分组倍率、计费设置里的专业术语。
- 后端 API 字段名，如 `quota`、`topup`、`billing_preference`。
- 数据库表名和 model 字段。
- 日志内部类型。
- 英文源码变量名。

否则会造成开发理解混乱，且没有降低用户侧法律风险的收益。

## 用户侧页面设计

### 钱包/支持中心页面总布局

页面仍作为用户购买订阅、兑换码和查看可用点数的入口，因此不能因为关闭钱包充值就隐藏整个入口。

正确规则：

- `subscription_purchase=true` 时，用户仍能进入该页面购买订阅。
- `wallet_topup=false` 时，只隐藏直接充值区域。
- `redemption_redeem=false` 时，只隐藏兑换码输入。
- 三者都关闭，且没有历史订阅可展示时，页面显示一个空状态，提示“当前暂无可用的自助支持入口，请联系管理员”。

### 充值区域

显示条件：

```text
wallet_topup=true 且至少一个 wallet_topup 支付通道可用
```

如果 `wallet_topup=false`：

- 不显示充值金额输入。
- 不显示充值挡位。
- 不显示支付方式按钮。
- 不请求 amount 试算接口。
- 不显示“在线充值未启用，请使用兑换码”的提示，除非兑换码仍开启且确实需要引导。

如果 `wallet_topup=true` 但没有任何通道可用：

- 可以显示“暂无可用支付方式，请联系管理员”。
- 这是配置异常提示，不是业务关闭提示。

### 兑换码区域

显示条件：

```text
redemption_redeem=true
```

如果关闭：

- 不显示兑换码输入。
- 不显示“兑换码功能已禁用”的大块警告，避免页面显得像出错。

### 订阅区域

显示条件：

```text
有套餐或用户有历史订阅
```

购买按钮启用条件：

```text
subscription_purchase=true 且至少一个 subscription_purchase 支付通道可用
```

如果 `subscription_purchase=false`：

- 可以保留已有订阅列表。
- 套餐购买按钮显示“暂不可购买”或直接隐藏。
- 不影响用户查看当前订阅、切换扣费偏好中仍合法的选项。

### 邀请面板

显示条件：

```text
billing_display_setting.invitation_panel_enabled=true
```

如果关闭：

- 旧版不渲染 `InvitationCard`。
- 旧版不调用 `getAffLink()`，避免无意义生成邀请码。
- 新版如果未来重新挂载 `AffiliateRewardsCard`，同样先判断开关，再调用 `useAffiliate`。
- 转移弹窗 `TransferModal` 不应从隐藏入口打开；如果 URL 或状态异常打开，也应安全关闭。

### 入口文案

公益性显示模式开启时：

- 新版侧边栏显示“支持中心”。
- 旧版侧边栏和用户下拉菜单显示“支持中心”。
- 页面标题显示“支持中心”。
- 充值卡片标题显示“项目支持”。
- 余额类显示“可用点数”。
- 模型扣费单位显示“模型点数”或“系统原生点数”。

## 与订阅设计的关系

订阅购买不是钱包充值，不应被 `wallet_topup=false` 影响。

需要特别保留：

- 套餐列表。
- 当前订阅和历史订阅。
- 订阅扣费偏好。
- 管理员给用户绑定订阅。
- 订阅支付回调。

需要避免：

- 用 `enable_online_topup` 判断订阅易支付是否可用。
- 用 `pay_methods` 作为订阅支付方式的唯一来源。
- 关闭钱包充值后让订阅购买按钮一起消失。

订阅组件应优先读取：

```text
payment_methods_by_scene.subscription_purchase
```

兼容读取：

```text
subscription_payment_methods
```

最后才回退到旧字段。

## 与邀请逻辑的关系

邀请面板关闭不等于邀请系统关闭。

本次建议范围：

- 关闭用户侧展示。
- 避免前端主动请求邀请码。
- 保留后端邀请码和注册链路兼容。
- 保留已有邀请收益数据。

不建议本次顺手做：

- 清空用户 `aff_code`。
- 禁止邀请链接注册。
- 清零 `aff_quota`。
- 删除邀请字段。

如果后续要进一步关闭邀请收益，应单独设计：

- 注册是否接受 `aff` 参数。
- 邀请人数是否继续统计。
- 邀请双方奖励是否继续发放。
- 既有 `aff_quota` 如何处理。
- 管理员是否能手动转移或清算。

## 与侧边栏开关的关系

现有 `SidebarModulesAdmin` 和用户个人侧边栏设置已经能隐藏 `personal.topup` 整个入口，但它不适合替代本次需求。

原因：

- 用户买订阅仍需要进入该页。
- `personal.topup=false` 会隐藏整页入口，用户找不到订阅入口。
- 本次要隐藏的是钱包充值、邀请面板、兑换码等局部能力，不是整个“支持中心”。

所以设计上：

- `SidebarModulesAdmin.personal.topup` 控制整页入口是否存在。
- `business_features.wallet_topup` 控制充值区域是否存在。
- `business_features.subscription_purchase` 控制订阅购买是否可自助购买。
- `billing_display_setting.public_welfare_text_enabled` 控制入口怎么命名。

四者职责不同，不能混用。

## 正确性分析

### 功能正确性

`wallet_topup=false` 后，用户不能自助钱包充值，因为后端已有 guard，前端也不再显示充值区域和触发试算。

`invitation_panel_enabled=false` 后，用户不再看到邀请面板，旧版不再请求邀请码，新版未来重新挂载邀请组件时也受同一开关控制。

- 分组倍率。
- 订阅扣费顺序。
- 支付订单和回调。
- 历史账单。

因此不会影响实际计费。

- 不启用公益性文案。
- 显示邀请面板。

### 安全边界正确性

前端隐藏只作为体验层。钱包充值、兑换码和订阅购买的核心允许/拒绝由后端 guard 决定。

## 上下设计冲突分析

### 冲突 1：支持中心入口是否应该隐藏

如果关闭钱包充值就隐藏整个入口，会导致用户无法购买订阅。

解决：入口不跟 `wallet_topup` 绑定，入口仍由侧边栏模块控制；页面内部按业务能力显示不同区域。

### 冲突 2：邀请面板开关是否应该放进支付能力

邀请收益确实和额度有关，但已有设计已经把邀请奖励从 `payment_setting.business_features` 收敛出去。

解决：本次只加 `invitation_panel_enabled` 作为显示开关，不改变奖励发放规则。

### 冲突 3：公益性文案是否影响管理端

管理端如果也把所有“额度/充值/计费”改成公益性文案，会让管理员配置模型价格、倍率、通道时更难理解。

解决：公益性显示只用于用户侧。管理端保留技术术语，只在设置项本身说明该模式的用户侧效果。

### 冲突 4：旧版和新版页面职责不同

新版当前没有实际显示邀请面板，旧版仍有。只按截图判断会遗漏旧版。

解决：方案要求两套前端都接入同一配置。新版即使当前不显示，也要保证未来挂载组件时受控。

### 冲突 5：`/api/user/topup/info` 既服务充值又服务订阅

如果用钱包充值字段判断订阅支付，会继续误伤。

解决：订阅优先读取 `payment_methods_by_scene.subscription_purchase` 和 `enable_*_subscription`。

## 潜在缺陷与规避

### 缺陷 1：文案 helper 覆盖不全

如果只改钱包页标题，其他入口仍显示“钱包/充值/余额”。

规避：开发清单中单独设置“文案扫描阶段”，用 `rg` 搜索关键词：

```text
钱包|充值|余额|额度|账户充值|钱包管理|优先钱包|仅用钱包
```

逐项判断是否属于用户侧并接入 helper。

### 缺陷 2：隐藏 UI 但仍发请求

旧版当前会固定请求 `/api/user/aff`。

规避：把获取邀请码逻辑放到 `invitationPanelEnabled` 条件后，关闭时不调用。

### 缺陷 3：关闭充值后显示空卡片

用户会以为系统异常或配置没开。

规避：如果 `wallet_topup=false` 且 `redemption_redeem=false`，不渲染充值卡片；如果订阅开启，只展示订阅区域。

### 缺陷 4：公益性模式污染默认翻译

硬改 `zh.json` 会影响所有中文用户。

规避：恢复通用词条语义，公益性词表只由 helper 在开关开启时返回。

### 缺陷 5：新配置没有默认值

升级后可能出现 undefined，导致页面显示不稳定。

规避：后端默认值、前端解析默认值都设置：

```text
public_welfare_text_enabled=false
invitation_panel_enabled=true
```

### 缺陷 6：乱码扩大

旧版部分文件在终端输出中已有乱码。如果继续复制乱码文本，会扩大问题。

规避：

- 新增文件和新增文案统一 UTF-8。
- 尽量用已有正常中文或英文 key。
- 修改后用 `Get-Content -Encoding UTF8` 回读新增文档和新增代码片段。
- 用 `rg -n "锛|鍏|绔|閭|璐"` 检查新增 diff，不把已有乱码当作本次新增问题。

可以后置：

- 更细的后端邀请系统禁用。
- 邀请奖励发放独立开关。
- 管理端更多文案随公益性模式变化。
- 独立 `/api/billing/display` 接口。

## 结论

本次不应继续用硬改翻译和硬隐藏组件解决问题。更稳的方案是：

- 保留已有 `payment_setting.business_features` 管用户侧支付/兑换能力。
- 新增 `billing_display_setting` 管用户侧显示策略。
- 公益性文案只在开关开启时作用于用户侧。
- 邀请面板用独立显示开关控制，并关闭对应前端请求。
- 支持中心入口不等同于钱包充值入口，关闭充值后仍保留订阅购买路径。

这样既能满足当前站点降低“钱包/充值/额度”展示风险的需求，也不会破坏订阅购买、上游默认行为和新旧前端一致性。
