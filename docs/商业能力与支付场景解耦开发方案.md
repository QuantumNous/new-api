# 商业能力与支付场景解耦开发方案

## 2026-05-19 设计修订

本方案原先把所有“会产生或转移额度”的入口都放进 `payment_setting.business_features`，实践后发现这会和已有业务配置产生重复控制，容易出现“一个地方开、另一个地方关”的状态冲突。

修订后的边界如下：

- `payment_setting.business_features` 只保留没有更准确独立配置的用户侧入口：`wallet_topup`、`subscription_purchase`、`redemption_redeem`。
- `redemption_manage` 不再作为支付页业务能力，后台兑换码入口仍由运营设置里的 `SidebarModulesAdmin.admin.redemption` 控制；创建/修改兑换码只保留合规确认校验。
- `invitation_reward` 不再作为支付页业务能力，邀请奖励是否产生由 `QuotaForInviter`、`QuotaForInvitee` 和合规确认共同决定。
- `invitation_transfer` 不再作为支付页业务能力，用户是否能划转由已有邀请奖励余额和合规确认决定。
- `checkin_reward` 不再作为支付页业务能力，签到奖励由 `checkin_setting.enabled`、`checkin_setting.min_quota`、`checkin_setting.max_quota` 控制。
- 后端仍兼容旧 JSON 中的 `redemption_manage`、`invitation_reward`、`invitation_transfer`、`checkin_reward`，但保存时会清理掉，不再写回为有效配置。

以下历史方案中涉及上述四个旧业务能力键的设计，均以本修订为准。

## 目标

本方案要解决的不是单独隐藏“额度充值”按钮，而是把系统里三类概念拆开：

- 支付通道配置：易支付、Stripe、Creem、Waffo 等是否配置完整，是否能验签和拉起支付。
- 业务能力开关：用户是否允许额度充值、是否允许购买订阅、是否允许兑换码、是否允许邀请奖励等。
- 支付场景授权：某个支付通道是否允许用于某个业务场景，例如易支付只用于订阅套餐，不用于钱包额度充值。

当前运营目标可以通过该通用能力表达为：

- 关闭用户钱包额度充值。
- 保留订阅套餐购买和支付。
- 防止用户通过自定义金额、清空充值挡位后的手动输入、直接请求 API 等方式绕过。
- 后续如果还有其他“给用户增加余额/额度”的入口，也能统一看见、统一控制，而不是每次临时补洞。

## 问题分析

### 1. 清空充值挡位不能关闭充值

`payment_setting.amount_options` 只影响预设充值按钮。前端仍有自定义金额输入，后端也仍接受：

- `POST /api/user/amount`
- `POST /api/user/pay`
- `POST /api/user/stripe/amount`
- `POST /api/user/stripe/pay`

所以清空挡位不是安全方案。

### 2. 清空支付方式会误伤订阅

`PayMethods` 是易支付支付方式列表，钱包充值和订阅易支付都依赖它。清空后：

- 钱包易支付充值不可用。
- 订阅易支付购买也不可用。

这不符合“保留套餐支付”的目标。

### 3. 只隐藏前端不安全

即使前端隐藏充值区域，用户仍可以直接调用后端接口创建钱包充值订单。所以后端必须拦截创建订单和金额试算接口。

### 4. 回调不能按业务开关拦截

业务开关只应控制“创建新订单/发放新奖励”。已经创建的 pending 订单回调不能因为后来关闭开关而失败，否则会出现用户已付款但系统不入账。

正确规则：

- 创建新钱包充值订单：受业务能力和通道场景限制。
- 创建新订阅支付订单：受订阅购买能力和通道场景限制。
- 支付回调：只受通道配置、签名校验、订单状态、订单类型校验限制。

### 5. 订阅完成会写 TopUp 记录，不能按 TopUp 表名粗暴判断

`model/subscription.go` 的 `CompleteSubscriptionOrder` 会调用 `upsertSubscriptionTopUpTx`，在 `topups` 表写入一条金额为 0 的成功记录，用于统一展示订单/日志。

因此“TopUp 表出现记录”不一定代表钱包额度充值。后续统计或 UI 文案不能只按表名判断，需要按 trade_no 前缀、amount、provider、subscription_orders 是否存在等上下文区分。

### 6. 兑换码、邀请奖励、签到奖励都是额度入口

除了在线支付充值，还有其他能增加用户可消费余额/额度的入口：

- 兑换码：`model.Redeem`。
- 邀请奖励发放：`model.User.Insert`、`model.User.FinalizeOAuthUserCreation`。
- 邀请奖励转余额：`controller.TransferAffQuota`、`model.TransferAffQuotaToQuota`。
- 签到奖励：`controller.DoCheckin`、`model.UserCheckin`。
- 新用户赠送额度：`common.QuotaForNewUser`。
- 管理员改用户额度：`controller.ManageUser` 等管理操作。
- 退款/回补：任务失败退款、预扣退款、钱包资金源结算回补等。

这些不应该全部被“关闭充值”一刀切。需要明确分类：

- 用户主动购买型额度入口：钱包在线充值，应可关闭。
- 用户凭证型额度入口：兑换码，应可单独关闭用户兑换，但保留管理员生成能力可选。
- 用户营销奖励入口：邀请奖励、签到、新用户赠送，应有独立开关或继续由原有数值/功能开关控制。
- 管理员运维入口：管理员改额度、管理员绑定订阅，不应被用户侧商业能力开关影响。
- 退款和结算入口：不能被商业开关影响，否则会破坏账务一致性。

## 设计原则

1. 通道配置只表示“技术上能不能用”，不表示“业务上允不允许卖”。
2. 业务能力开关只管入口，不管已经产生的订单回调、退款、补偿、管理员运维。
3. 支付场景授权按 provider + scene 判断，不能按 provider 全局一刀切。
4. 前端隐藏只是体验优化，后端拦截才是最终边界。
5. 新字段默认保持兼容，升级后不能让现有站点突然失效。
6. 旧 API 可以保留兼容字段，但新增能力接口要提供清晰语义。
7. 所有配置都走 `Option` / `GlobalConfig` 体系，不直接改数据库结构，除非确实需要订单层数据。
8. 文案和字段名保持一致，不引入含糊词：`feature` 表示业务能力，`scene` 表示支付使用场景，`provider` 表示支付通道。

## 推荐配置模型

在 `setting/operation_setting/payment_setting.go` 的 `PaymentSetting` 中新增两个结构化字段。

```go
type PaymentSetting struct {
    AmountOptions []int `json:"amount_options"`
    AmountDiscount map[int]float64 `json:"amount_discount"`

    ComplianceConfirmed bool `json:"compliance_confirmed"`
    ComplianceTermsVersion string `json:"compliance_terms_version"`
    ComplianceConfirmedAt int64 `json:"compliance_confirmed_at"`
    ComplianceConfirmedBy int `json:"compliance_confirmed_by"`
    ComplianceConfirmedIP string `json:"compliance_confirmed_ip"`

    BusinessFeatures map[string]bool `json:"business_features"`
    ProviderSceneScopes map[string]map[string]bool `json:"provider_scene_scopes"`
}
```

建议默认值：

```go
BusinessFeatures: map[string]bool{
    "wallet_topup": true,
    "subscription_purchase": true,
    "redemption_redeem": true,
    "redemption_manage": true,
    "invitation_reward": true,
    "invitation_transfer": true,
    "checkin_reward": true,
},
ProviderSceneScopes: map[string]map[string]bool{
    "epay": {
        "wallet_topup": true,
        "subscription_purchase": true,
    },
    "stripe": {
        "wallet_topup": true,
        "subscription_purchase": true,
    },
    "creem": {
        "wallet_topup": true,
        "subscription_purchase": true,
    },
    "waffo": {
        "wallet_topup": true,
    },
    "waffo_pancake": {
        "wallet_topup": true,
    },

```

### 命名说明

业务能力：

- `wallet_topup`：用户在线钱包额度充值。
- `subscription_purchase`：用户自助购买订阅套餐。
- `redemption_redeem`：用户使用兑换码。
- `redemption_manage`：管理员创建、修改、启用/禁用兑换码。
- `invitation_reward`：新用户注册后给邀请双方产生奖励。
- `invitation_transfer`：用户把已获得的邀请码奖励转入钱包余额。
- `checkin_reward`：用户签到获得额度。

支付场景：

- `wallet_topup`
- `subscription_purchase`

支付通道：

- `epay`
- `stripe`
- `creem`
- `waffo`
- `waffo_pancake`

### 为什么使用 map 而不是多个 bool 字段

优点：

- 新增通道或业务场景时不需要频繁改配置结构。
- 前端可以用矩阵渲染，不需要每个通道写死一套字段。
- 和 `GlobalConfig` 的 JSON 配置模式兼容。

风险：

- map 的 key 拼写错误不容易被编译器发现。

规避：

- 后端定义常量。
- 加 normalize 函数补默认值。
- 保存配置时校验 key，只允许已知 feature、scene、provider。
- 测试覆盖未知 key、缺失 key、旧配置升级。

## 通道可用性重构

当前函数名带 `TopUp`，建议分层重命名或保留兼容包装。

底层通道配置判断：

```go
func isEpayConfigured() bool
func isStripeConfiguredForCheckout() bool
func isStripeWebhookConfigured() bool
func isCreemConfiguredForCheckout() bool
func isCreemWebhookConfigured() bool
func isWaffoConfigured() bool
func isWaffoPancakeConfigured() bool
```

业务场景判断：

```go
func isEpayAvailableForScene(scene string) bool {
    return isPaymentComplianceConfirmed() &&
        isEpayConfigured() &&
        operation_setting.IsPaymentProviderAllowedForScene("epay", scene)
}
```

旧函数兼容：

```go
func isEpayTopUpEnabled() bool {
    return isEpayAvailableForScene(operation_setting.PaymentSceneWalletTopUp)
}
```

新增订阅场景函数：

```go
func isEpaySubscriptionEnabled() bool
func isStripeSubscriptionEnabled() bool
func isCreemSubscriptionEnabled() bool
```

Webhook 判断不要使用业务场景函数：

```go
func isStripeWebhookEnabled() bool {
    return isPaymentComplianceConfirmed() && isStripeWebhookConfigured()
}
```

如果担心未启用任何 Stripe 业务但 webhook 暴露，可以保留“通道配置完整 + 合规确认”判断，不要绑定 `wallet_topup`。

## 后端拦截点

### 钱包在线充值

以下接口必须同时检查：

- `payment_setting.business_features.wallet_topup`
- 通道是否配置完整。
- provider 是否允许 `wallet_topup` 场景。

接口：

- `controller.RequestAmount`
- `controller.RequestEpay`
- `controller.RequestStripeAmount`
- `controller.RequestStripePay`
- `controller.RequestCreemPay`
- `controller.RequestWaffoAmount`
- `controller.RequestWaffoPay`
- `controller.RequestWaffoPancakeAmount`
- `controller.RequestWaffoPancakePay`

其中 `amount` 试算接口也必须拦，否则前端虽然不能支付但仍泄露可用性，且容易给绕过者提示。

### 订阅购买

以下接口必须检查：

- `payment_setting.business_features.subscription_purchase`
- provider 是否允许 `subscription_purchase` 场景。
- 原有套餐启用、限购、plan payment id 等校验继续保留。

接口：

- `controller.SubscriptionRequestEpay`
- `controller.SubscriptionRequestStripePay`
- `controller.SubscriptionRequestCreemPay`

管理员手动绑定订阅：

- `controller.AdminBindSubscription`
- `controller.AdminCreateUserSubscription`

建议不受 `subscription_purchase` 用户自助购买开关限制。管理员运维和用户自助购买是两种能力。如果需要控制，可另设 `subscription_admin_manage`，不要混用。

### 兑换码

用户兑换：

- `controller.TopUp`
- 检查 `redemption_redeem`。

管理员管理：

- `controller.AddRedemption`
- `controller.UpdateRedemption`
- `controller.DeleteRedemption`
- `controller.DeleteInvalidRedemption`

建议只对创建/修改状态检查 `redemption_manage`，查询和删除可保留管理员能力，避免管理员无法清理历史数据。

### 邀请奖励

奖励产生位置：

- `model.User.Insert`
- `model.User.FinalizeOAuthUserCreation`

检查 `invitation_reward`。关闭后：

- 仍允许用户使用邀请链接注册。
- 不给 invitee 增加 `QuotaForInvitee`。
- 不给 inviter 增加 `AffQuota` / `AffHistoryQuota` / `AffCount` 是否增加需明确。

建议：

- `AffCount` 可以继续增加，代表邀请人数。
- `AffQuota` 和 `AffHistoryQuota` 不增加。
- 日志不写“赠送额度”。

邀请奖励转余额：

- `controller.TransferAffQuota`
- 检查 `invitation_transfer`。

关闭后：

- 已有 `aff_quota` 保留。
- 用户不能转入钱包余额。
- 管理员仍可手动处理用户额度。

### 签到奖励

当前签到已有独立配置 `checkin_setting.enabled`。如果要统一商业能力，可以让 `checkin_reward` 和 `checkin_setting.enabled` 同时为 true 才允许：

- `controller.GetCheckinStatus`
- `controller.DoCheckin`
- `model.UserCheckin`

为了避免重复开关造成困惑，建议第一期只在能力接口里展示 check-in 状态，不改签到逻辑。第二期如确实需要再统一。

### 不应该拦截的逻辑

以下逻辑不能被用户侧商业能力开关拦截：

- 支付 webhook 完成历史 pending 订单。
- 订单过期、失败状态回写。
- 管理员手动补单。
- 管理员改用户额度。
- 任务失败退款。
- 预扣退款。
- 订阅消费退款。
- `service` 层账务结算的 delta 回补。

原因：这些属于账务一致性或管理员运维，不是新业务销售入口。

## 风险分析

### 风险 1：误伤支付回调

如果把业务开关加入 `isStripeWebhookEnabled()` 等函数，会导致历史订单回调失败。

规避：

- webhook 判断只检查合规和通道验签配置。
- 创建订单才检查业务能力和场景授权。

### 风险 2：订阅购买仍依赖 topup info

前端订阅弹窗现在读取 `enableOnlineTopUp` 等字段。如果这些旧字段改成钱包充值语义后，订阅按钮可能消失。

规避：

- 新增 `payment_methods_by_scene.subscription_purchase`。
- 订阅组件优先使用新字段。
- 过渡期保留旧字段但只给钱包充值使用。

### 风险 3：只拦 pay，不拦 amount

用户仍可试算金额，前端状态混乱，也暴露通道可用性。

规避：

- pay 和 amount 接口都加同样的场景判断。

### 风险 4：兑换码和邀请奖励遗漏

用户仍可能通过非支付路径增加钱包额度。

规避：

- 文档和开发清单明确覆盖 `TopUp`、`TransferAffQuota`、邀请奖励发放。
- 签到和新用户赠送作为相关但不默认拦截的入口在自检中确认。

### 风险 5：管理员运维被错误限制

管理员绑定订阅、改额度、补单如果被拦，会影响售后。

规避：

- 用户自助能力和管理员运维能力分开。
- 默认不限制管理员修复型操作。

### 风险 6：前端两套界面不一致

项目有 classic 和 default 两套前端。如果只改一套，用户可能从另一套入口绕过体验限制。

规避：

- 两套前端都改。
- 后端强拦截保证最终安全。

### 风险 7：中文乱码

当前终端中部分源码中文显示为乱码。继续写入这些文件时可能扩大乱码。

规避：

- 优先用英文 key 和少量已有 i18n 文案。
- 新增文案走项目现有 i18n 机制。
- 保存前用编辑器确认 UTF-8。
- 自检时搜索明显乱码片段，如 `锛`、`绔`、`鍏` 是否来自新增代码。已有乱码不在本功能中扩大。

## 方案自身审查

### 是否符合需求

符合。该方案不是单点开关，而是通用能力体系，可以表达：

- 关闭钱包充值但保留订阅支付。
- 某个支付通道只用于订阅，不用于充值。
- 兑换码、邀请奖励、签到等额度入口可被单独纳入策略。

### 是否完善

基本完善，覆盖了：

- 后端路由入口。
- 支付通道可用性。
- 支付回调边界。
- 前端两套 UI。
- 后台配置。
- 兼容旧配置。
- 管理员运维和退款边界。

仍需在实施阶段继续确认：

- Waffo 是否未来会支持订阅场景。如果会，provider-scopes 要允许增加 `subscription_purchase`。
- Creem 产品配置和订阅 plan 的 `creem_product_id` 是否存在业务重叠。
- 现有 `TopUpHistory` 对订阅订单的展示是否需要随新语义调整。

### 是否完整

对“支付售卖入口”和“额度增加入口”覆盖较完整。没有把所有 `IncreaseUserQuota` 调用都纳入拦截，这是有意设计，因为退款、结算、管理员操作不能被销售开关影响。

### 是否正确

正确性关键点：

- 创建订单前拦截。
- 回调不按业务开关拦截。
- 订阅支付不再依赖钱包充值字段。
- 旧配置缺失默认全开，避免升级破坏。

### 是否规范一致

与项目现有规范一致：

- 配置继续走 `payment_setting.*` 和 `GlobalConfig`。
- 后端仍用 controller + model 分层。
- 前端 default 继续走 system-settings section registry。
- classic 继续走 `PaymentSetting.jsx` tab 结构。
- API 返回保持 `success/message/data` 或现有支付接口 `message: success` 风格。

### 是否存在上下设计冲突

潜在冲突 1：`payment_setting.compliance_confirmed` 当前被当作总开关。

解决：合规确认仍是总前置条件；业务能力开关在合规确认之后生效。即：

```text
合规未确认：全部商业能力关闭。
合规已确认：再看 business_features 和 provider_scene_scopes。
```

潜在冲突 2：旧字段名 `enable_online_topup` 被订阅弹窗使用。

解决：保留旧字段，但新增按 scene 返回的新字段，逐步迁移订阅弹窗。

潜在冲突 3：订阅完成写入 TopUp 记录。

解决：不按 TopUp 表名判断业务入口；按创建订单接口和 subscription_orders 区分。

潜在冲突 4：邀请奖励和邀请转余额是否同属一个功能。

解决：拆成 `invitation_reward` 和 `invitation_transfer`。一个控制新奖励产生，一个控制已有奖励转余额。

### 阶段 8：人工验收场景

场景 A：钱包充值关闭，订阅开启。

- `/api/user/amount` 拒绝。
- `/api/user/pay` 拒绝。
- `/api/user/stripe/pay` 拒绝。
- `/api/subscription/plans` 返回套餐。
- `/api/subscription/epay/pay` 可创建订单。
- `/api/subscription/stripe/pay` 可创建订单。
- 前端钱包充值区域隐藏。
- 前端套餐购买按钮存在。

场景 B：订阅关闭，钱包充值开启。

- 钱包充值可用。
- 套餐列表可显示或显示不可购买状态。
- `/api/subscription/*/pay` 拒绝。

场景 C：易支付只允许订阅。

- `provider_scene_scopes.epay.wallet_topup=false`
- `provider_scene_scopes.epay.subscription_purchase=true`
- `/api/user/pay` 拒绝。
- `/api/subscription/epay/pay` 可用。

场景 D：关闭兑换码兑换但保留管理。

- 用户 `/api/user/topup` 拒绝。
- 管理员能查询兑换码。
- 管理员能删除历史兑换码。
- 如 `redemption_manage=true`，管理员能创建兑换码。

场景 E：历史订单回调。

- 创建 pending 订单后关闭对应业务能力。
- 模拟回调。
- 订单仍能完成或过期。
- 不出现用户已支付但无法入账。

## 实施顺序建议

推荐顺序：

1. 后端配置模型。
2. 后端通道和场景判断。
3. 后端拦截。
4. 能力接口。
5. Default 前端。
6. Classic 前端。
7. 测试和人工验收。

不要先只做前端隐藏。前端隐藏没有安全边界，容易造成“看起来关了，其实 API 还能充”的错觉。

## 最小可上线版本

如果要压缩范围，最小版本也必须包含：

- `business_features.wallet_topup`
- `business_features.subscription_purchase`
- `provider_scene_scopes`
- 钱包充值 pay/amount 后端拦截
- 订阅 pay 后端保留
- webhook 不受业务开关影响
- Default 和 Classic 的充值 UI 隐藏

可后置：

- 邀请奖励开关
- 签到奖励纳入能力接口
- 完整的 `/api/billing/capabilities`
- 后台矩阵可视化编辑器

但不建议后置兑换码，因为兑换码同样是钱包额度入口，容易和“关闭额度充值”的运营目标冲突。

## 结论

最稳妥的设计是“商业能力 + 支付场景矩阵”。它能解决当前要关闭额度充值但保留套餐支付的问题，也能避免以后新增支付通道、兑换码、邀请奖励、签到奖励时再次出现配置互相牵连。

实现时必须坚持：

- 配置通道不等于开放业务。
- 业务开关不拦历史回调。
- 用户入口和管理员运维分开。
- 后端拦截优先，前端隐藏跟进。
- 每个阶段开发后都按清单自检，避免遗漏边界和新增乱码。
